# 第 13 章 失败模式与边界

## 13.1 常见的 PM 失败模式

**过度自信于 AI 生成的解释。** Claude Code 阅读代码并提供解释，但它无法运行代码。其解释是高置信度假设，而非绝对真理。当准确性重要时（面向客户的沟通、路线图决策、安全评估），在行动前与工程团队验证。

**将 Claude Code 用于错误的工具。** 如果你在用 Claude Code 起草邮件、头脑风暴概念或获取快速答案，你正在错误的地方。Claude.ai 更适合这些。使用 Claude Code 做它设计用来做的事：基于文件的调查、工件生成、可重复工作流。

**建立过于复杂的技能。** 从简单开始。一个能做好一件事的 30 行技能比一个试图覆盖所有边缘情况但却失败的 300 行怪物更有价值。先构建最小可行技能，基于使用迭代。

**在数据准备上偷工减料。** Claude Code 合成你给它的东西。垃圾进，垃圾出。在要求合成之前，花时间清洗反馈数据、正确格式化文件并组织研究材料。

**忽视成本，然后感到震惊。** 作为 API 用户，长时间会话累积起来很快。每会话后检查 `/cost`。设置预算警报。如果成本过高，使用 `/compact`。作为订阅用户，不要忽视 token 消耗——高消耗意味着更慢的响应和更笨重的会话。

## 13.2 知道何时已提取了可用价值

**对调查的递减回报**（第 4.6 节进行了扩展）：前 20 分钟产生 80% 的价值。此后，除非你在探索全新的方面，否则继续迭代会支付递减回报。

**对技能构建的递减回报**：一旦技能对相同输入始终产生一致输出，额外的调整产生递减回报。停止调整，开始使用。

## 13.3 属于工程团队的任务

- **提交代码变更**。你可能调查并建议，但只有工程团队可以提交。
- **运行数据库迁移**。需要生产访问权限，超出 PM 范围。
- **做出架构决策**。Claude Code 可以告知，但不能决定。
- **安全审查**。需要只有工程师拥有的专业知识。
- **调试竞态条件**。需要运行时观察，而非静态代码分析。

Claude Code 赋予你理解权——而非修改或部署权。尊重该边界。

## 13.4 让 Claude Code 长期有效的习惯

1. **维护你的 CLAUDE.md。** 当领域知识变化时更新它。这是你的 AI 队友的入职文档，最好保持更新。
2. **使用 `/cost` 培养直觉。** 经过 10-15 个会话，你将凭直觉知道哪些类型的任务消耗 token。
3. **默认为 plan 模式。** 除非有意创建或修改文件，否则在只读模式下启动。安全默认设置防止事故。
4. **定期审查和刷新技能。** 每个季度，对每个技能进行一次测试运行。更新任何已失效的内容。
5. **记录你的发现。** 每次调查都应产出一个工件——一份文档，存在于仓库中。使制度知识显式化。
6. **在不确定时升级给工程团队。** 当调查遇到第 3.6 节中的任何边界时，交接并透明地说明你尝试了什么。
7. **使用适合每个任务的工具。** Claude.ai 用于思考，Claude Code 用于执行。两者都使用，有意识地使用。

Claude Code 是一款放大 PM 效能的工具——而非替代工程判断。有效使用它的 PM 会成为更好的合作伙伴：信息更充分，准备更充分，更尊重工程团队的时间。使用得当，它改变游戏规则。本书已向你展示了如何做到。
