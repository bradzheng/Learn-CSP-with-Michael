# 附录 D：提示词模板

复制并适配这些提示词用于常见 PM 工作流。`[方括号]` 中的占位符需要你填入具体内容。详细上下文和示例见各引用章节。

## 调查类提示词

### 架构概览
```
解释此代码库的架构，面向产品经理。涵盖：
- 主要组件及各自职责
- 数据如何在系统中流转
- 面向用户的功能通常在哪里
- 前端和后端如何连接（如适用）
- 我应该理解的关键抽象或模式
保持解释非技术化。
```

### 功能实现
```
[功能] 在此应用中如何工作？具体来说：
- [功能] 逻辑在哪里实现？
- 什么规则/约束控制其行为？
- [你应该知道的相关细节]
用产品术语解释，而非代码细节。
```

### 数据访问
```
[功能] 访问什么用户数据？展示：
- 读取了哪些数据库表或 API 端点
- 使用了哪些字段
- 是否回写数据
- 我应该知道的任何隐私或安全考虑
```

### 边缘情况调查
```
当 [条件] 时会发生什么？带我走过：
- 用户看到什么（错误消息、页面状态）
- 系统内部发生什么
- 我们如何处理失败或异常
- 是否有缺口导致意外行为
```

### 行为解释
```
用户报告 [行为]。为什么会出现这种情况？
解释代码中什么导致此行为，以及是否有产品上的原因。
是设计如此还是 bug？
```

### Bug 分类
```
我正在调查一个用户报告的问题：[一句话摘要]
上下文：
- 用户操作：[他们在做什么]
- 预期：[应该发生什么]
- 实际：[实际发生了什么]
- 错误消息（如有）：[精确文本]
识别可能涉及的代码区域。请展示每个区域的入口点、执行路径中的关键文件、错误在哪里被捕获和处理、可能在哪里失败。
```

## 分析类提示词

### 主题识别
```
读取 [反馈文件夹] 中的所有反馈，识别前 10-15 个主题。
对每个主题提供：主题名称、描述（1-2 句）、频率（大致百分比）、严重性（高/中/低）、代表性引用（2-3 条原话）。
```

### 访谈综合
```
分析 [访谈文件夹] 中的所有访谈记录。提取：
- 跨访谈的关键主题（附有频率和引用）
- 痛点（用户挣扎的具体任务或工作流）
- 未满足的需求（用户表达的愿望）
- 让你惊讶的意外发现
```

### 竞品档案
```
使用网络搜索研究 [竞品名称]，在 research/competitors/[name].md 创建档案。
包括：公司概览、目标市场、核心功能、定价模式、优势、劣势、近期动态、最后更新日期。
```

### 趋势综合
```
审查 [趋势文件夹] 中的所有来源，将关键主题综合为一个趋势简报。
对每个主要趋势：趋势名称、描述（2-3 句）、支持证据（引用具体文件）、时间（现在/即将/将来）、对我们的相关性（高/中/低）、影响（2-3 条我们应该考虑做的事情）。
```

### 市场估算
```
使用 TAM/SAM/SOM 框架帮我估算 [产品/类别] 的市场规模。
对每个指标提供：定义、计算方法、数据来源、假设、估算范围（低/中/高）、推理过程。
展示工作过程。使假设明确。
```

## 生成类提示词

### PRD 生成
```
从 [用户故事文件] 创建 PRD，引用代码库约束。
包括：问题陈述、成功指标、用户故事、技术考虑、范围（包含/排除）、开放问题、假设。
引用此代码库中现有的实现模式。
```

### 画像生成
```
基于 [研究文件夹] 中的研究材料，创建 [3-5] 个用户画像，保存到 docs/personas/。
对每个画像包括：姓名和角色、目标与动机、痛点、行为模式、来自访谈的引用（如可用）。
```

### 变更日志
```
审查自上次发布以来的 git 历史。生成按以下分类的发布变更日志：
新功能、改进、Bug 修复、破坏性变更。用用户利益（而非技术细节）撰写。
```

### Bug 报告
```
基于我的调查，生成包括以下内容的 bug 报告：
问题摘要、复现步骤、环境细节、技术上下文（涉及的文件和代码路径）、我对根本原因的假设、建议工程团队的调查点、我无法回答的开放问题。
```

## 快速定制指南

使提示词更有效：
- **具体化**："认证如何工作？"→"当用户访问受保护页面时，应用如何确定用户是否登录？"
- **请求产品语言**："用产品术语解释"或"聚焦用户行为"
- **指定格式**："以表格形式呈现"、"包含 2-3 条引用"、"以编号步骤书写"
- **设定质量标准**："每个主题必须有 2+ 条支持引用"、"列出你的假设"、"评估你对每个发现的信心"
- **串联提示词**用于复杂工作流：1) 运行调查提示词收集上下文 → 2) 运行分析提示词综合发现 → 3) 运行生成提示词创建工件
