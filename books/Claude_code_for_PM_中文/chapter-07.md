# 第 7 章 需求、用户画像与文档

## 7.1 从分散研究中创建用户画像

你有来自 12 次客户访谈的笔记、去年的人口统计调查数据、显示功能使用模式的分析数据以及请求企业功能的支持工单。你需要用户画像，但研究分散在文档、笔记和电子表格中。Claude Code 将它们综合为结构化画像。

**图 7.1：画像生成工作流** ——将分散研究转化为结构化、可版本化的画像文档。研究来源（访谈、分析、支持数据）经过 Claude Code 综合成 `docs/personas/` 中的画像文件。

**准备研究材料。** 创建一个结构化目录：

```
research/persona-development/
├── interviews-enterprise-users.txt
├── analytics-summary.md
├── support-ticket-themes.md
└── survey-results.csv
```

> **提示词：用户画像生成**
>
> 分析 `research/persona-development/` 中的所有研究，创建 3-5 个用户画像。对每个画像包括：姓名和角色、人口统计（从数据中推导）、目标与动机、痛点与挫败感、行为模式、引用（来自访谈的真实引用，如可用）、与产品的典型交互。
>
> 将画像保存到 `docs/personas/` 作为 markdown 文件。

**画像结构**（Claude Code 生成）：
- **企业管理员**：做出购买决策，关心安全、集成、合规。证据：12 通销售电话中 8 通聚焦于企业需求。
- **终端用户**：个人贡献者，关注易用性和日常工作流效率，代表 70% 的用户。证据：支持工单和调查回复。
- **决策权威**（购买者、影响者、终端用户）
- **目标和动机**（他们试图实现什么）
- **痛点**（工作流中的具体摩擦点）

洞察优于人口统计细节。Claude Code 生成 `docs/personas/` 中的画像文件。每个画像成为参考文档，你可以从 PRD、路线图文档和用户故事中链接。

**成本**：创建 3-5 个画像从 5-10 个来源文档：20-30 分钟，30,000-60,000 token，$0.15-0.30。

## 7.2 从功能概念生成全面故事

你有一个功能概念："批量用户导入"。工程师需要用户故事、验收标准和边缘情况。传统上你会自己写这些。Claude Code 以系统化的方式生成。

> **提示词：用户故事生成**
>
> 为批量用户导入功能创建用户故事。从 `docs/personas/enterprise-admin.md` 中企业管理员画像的视角出发。
>
> 核心工作流：1. 管理员准备带有用户数据的 CSV 2. 系统验证并导入用户 3. 管理员审查结果。
>
> 包括：主要用户故事（"作为 [角色]，我想 [目标]，以便 [收益]"）、验收标准（可测试的条件）、边缘情况以及备注/约束。

Claude Code 参考画像以获取上下文、参考代码库以获取技术约束（例如，"导入文件必须小于 10MB，因为当前上传限制在 `config/uploads.js` 中定义"）。

输出：带编号步骤的全面用户故事、每个故事的验收标准、现实边缘情况（空白文件、超过 5,000 名用户、格式错误的 CSV、特殊字符处理、重复条目）。

**成本**：从功能概念生成故事：15-25 分钟，25,000-50,000 token，$0.13-0.25。

## 7.3 撰写保持最新的 PRD

PRD 需要引用代码库现实以保持持久价值。大多数 PRD 几乎立即过时，因为它们是在不了解实现的情况下写成的——或者因为没有人用代码变更来更新它们。Claude Code 生成引用实际文件并可随时间刷新的 PRD。

> **提示词：生成 PRD**
>
> 从 `docs/user-stories/bulk-user-import.md` 中的用户故事创建 PRD，引用代码库约束。包括：问题陈述、成功指标、用户故事、技术约束（引用相关文件）、范围（包含/排除）、开放问题和假设。
>
> 将 PRD 保存到 `docs/prds/bulk-user-import.md`。

**PRD 审计**：生成后，要求 Claude Code 根据你的团队标准审计它：

> 审计 `docs/prds/bulk-user-import.md`。验证：所有用户故事都有验收标准吗？技术约束引用实际文件吗？开放问题是具体的还是模糊的？范围部分明确区分了包含/排除吗？

**保持 PRD 最新**：随着实现推进，运行审计以检查 PRD 是否仍然准确。Claude Code 读取当前代码并比较："`config/uploads.js` 中的文件大小限制已从 10MB 更新为 25MB。PRD 仍显示 10MB。需要更新。"

**成本**：PRD 生成：15-25 分钟。PRD 审计：5-10 分钟。定期更新：5-10 分钟。

## 7.4 保持产品文档与实现同步

文档腐烂是普遍规律。API 文档、用户指南、变更日志——它们过时是因为维护它们需要额外的精力。Claude Code 直接从代码库生成或验证文档。

**变更日志生成。** 从 git 历史中生成发布说明：

> **提示词：变更日志**
>
> 审查自上次发布以来所有合并的拉取请求。生成发布变更日志，按以下分类：新功能、改进、Bug 修复、破坏性变更。对每个条目包括：变更描述（用户角度）、相关工单 ID（如存在）、致谢。保存为 `docs/changelog/v2.4.0.md`。

**用户指南生成。** 请求 Claude Code 从实际代码行为生成用户文档：

> **提示词：用户指南**
>
> 为导出功能创建用户指南。读取实际实现以了解工作原理，然后编写涵盖以下内容的文档：如何导出数据、支持哪些格式、导出中包含什么数据、限制和配额、导出大文件时会发生什么。保存到 `docs/guides/export-guide.md`。

**文档审计。** 检查现有文档是否与实际实现匹配：

> **提示词：文档审计**
>
> 比较 `docs/guides/export-guide.md` 与 `src/services/export.js` 的实际实现。标记任何差异：指南中描述但不再存在的功能、指南中未提及的已实现功能、指南中错误的限制或默认值。

**成本**：变更日志生成：10-15 分钟，$0.05-0.10。用户指南创建：20-30 分钟，$0.15-0.30。文档审计：15-25 分钟，$0.10-0.20。
