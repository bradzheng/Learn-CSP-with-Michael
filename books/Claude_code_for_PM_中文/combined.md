# 第 1 章 为什么 PM 要用 Claude Code——而不是 Claude.ai

## 1.1 选择合适的工具，每周节省数小时

一位客户报告了你的结账流程中有一个关键 bug。工程团队正全力投入季度路线图。你需要判断优先级：是真正的 bug 还是用户操作失误？影响范围有多大？归哪个团队负责？你等待工程团队切换上下文、调查并反馈。两天过去了，bug 仍然处于待分类状态。

或者：你在代码仓库中打开 Claude Code，问"结账验证是如何工作的？请告诉我 null 支付方式可能在哪些地方遗漏检查"，10 分钟内就得到了带有简明英文解释的文件引用。你整理出包含具体代码位置的调查报告。工程团队确认后直接修复，跳过了发现阶段。总耗时：15 分钟。

这两种场景的区别在于工具选择。大多数 PM 默认对所有事情使用 Claude.ai，因为它更熟悉，然后困惑为什么 AI 没有改变他们的工作流程。网页端处理对话式任务很好，但在 PM 最重要的工作上却力不从心：调查代码库、生成存放于代码仓库的产物、以及构建可复现的流程。

你访问 Claude 有三种方式：Claude.ai 网页界面、API 和 Claude Code。API 是给工程师构建集成用的。Claude.ai 和 Claude Code 都适合 PM，但它们解决不同的问题。选错的代价很高。

Claude Code 是一个具有文件系统访问权限、shell 命令执行能力以及跨会话持久化上下文能力的智能体（agent）。当你的工作涉及文件、代码仓库或可重复流程时，这些能力至关重要——无论你是否专业编程。

选错工具的隐性成本如下。你在 Claude.ai 上花了 30 分钟解释你的代码库背景，让它分析存储在 CSV 中的客户反馈，并请求生成综合文档。它给了你一个深思熟虑的回答。你复制粘贴到 Google 文档中。两周后，你需要对新反馈运行同样的分析。你从零开始：同样的 30 分钟背景说明，同样的手动导出，同样的复制粘贴。你在这个任务上已经花了一个小时，而 Claude Code 使用可复用技能只需五分钟即可完成。

**图 1.1：工作流对比** ——用于理解错误工具选择的隐性成本。左：手动复制粘贴工作流，每次重复上下文、手动导出和重新格式化。右：基于技能的自动化，设置只需一次，后续运行每次 5 分钟 vs 30 分钟。

或者你需要了解结账流程是如何工作的，因为收到了一个关键 bug 报告。你去问工程团队，但他们正在做冲刺规划。你尝试用 Claude.ai，但你无法上传整个代码库，没有上下文，它的回答都是泛泛之谈。你只能等待。工程团队明天才能回复你。bug 又多停留了一天。如果你用了 Claude Code，10 分钟内就能得到一个根本原因假设，并附有具体文件和函数的引用。

工具选择问题的核心在于任务持久性和数据访问。当工作存在于文件中、涉及迭代或发生不止一次时，网页端会强加手动开销，且随着每次重复而累积。Claude Code 通过直接在你的工作区中运行来消除这种开销。

大多数 PM 通过浪费才发现这一点。他们使用 Claude.ai 数周后才意识到，自己一直在向每个对话复制相同的上下文、手动重新格式化输出、重复做本该自动完成的工作。切换成本看似很高：安装、学习命令、理解权限模式。但浪费的成本更高，而且它在无声地累积。

这个决定与技术能力无关。它关乎你的工作是否涉及你会反复接触的文件，还是这只是一个一次性问题。诚实地回答，工具选择不言自明。

## 1.2 Claude Code 如何工作：自主性优于对话

Claude Code 是一个智能体运行时（agentic runtime）。这是技术术语。对产品经理来说，这意味着 Claude 可以行动而不仅仅是回应。当你问 Claude.ai 一个问题，它思考并回答。当你问 Claude Code 一个问题，它思考、读取文件、运行命令、生成产物、检查工作，然后回答。区别在于自主性。

这很重要，因为 PM 的工作很大程度上是跨分散来源的信息综合。需求在 Jira 中，客户反馈在 Zendesk 导出中，实现细节在代码库中，市场调研在收藏的 URL 和杂乱的笔记中。获得一个连贯的答案需要从所有这些来源中提取信息，手动完成会耗费数小时。

Claude Code 在你的项目目录中运行，拥有 Claude.ai 根本不具备的五项能力。理解这些能力，就能明白什么时候你需要的是智能体，而不仅仅是对话。

**图 1.2：五项核心能力** ——用于理解 Claude Code 能做到而 Claude.ai 不能的事情。文件系统读写访问、shell 命令执行、通过 CLAUDE.md 持久化项目上下文、用于并行工作的子智能体（subagent）生成、以及通过 MCP（模型上下文协议）连接外部系统。这些能力支撑了代码库调查、产物生成和可复现工作流。

**文件系统读写访问。** Claude Code 可以打开项目中的任何文件，读取、理解，并写入新文件或修改现有文件。这听起来很基础，直到你意识到 Claude.ai 需要你复制粘贴所有内容。需要在分析反馈时参考 PRD？用 Claude.ai，你要粘贴 PRD、粘贴反馈、请求综合，然后手动保存输出。用 Claude Code，你只需指向 `docs/prd.md` 和 `data/feedback.csv`，它就会直接在代码仓库中生成 `analysis/synthesis.md`。无需剪贴板，无需手动格式化，没有粘贴错误的风险。

对 PM 来说，这意味着调查输出存放在它们该在的地方：在版本控制中，与它们引用的代码并肩，对团队可见。你的代码库分析成为一个 markdown 文件，工程团队可以阅读、评论和更新。

**Shell 命令执行。** Claude Code 可以运行终端命令。对 PM 来说，这主要意味着 git 操作（查看历史记录、比较分支、查看谁改了什么）以及运行那些你不会自己编写但能说明需求的分析脚本。

具体例子：你需要了解某个特定功能是何时引入的、谁参与开发的。你可以请工程团队运行 `git log` 并解读结果，也可以直接问 Claude Code："结账流程的两步验证是什么时候添加的？当时的设计考虑是什么？"它会运行 git 命令，读取相关提交和文件变更，然后告诉你整个故事。无需打扰工程团队。

这使得终端能力通过自然语言变得可用，无需专业知识。

**通过 CLAUDE.md 持久化项目上下文。** 每次打开 Claude.ai，你都从零开始。每次在项目目录中启动 Claude Code，如果存在 CLAUDE.md 文件，它会自动读取：这个文件包含跨所有会话持久化的项目特定上下文。

对 PM 来说，这个文件可能包含产品领域术语表、映射到代码区域的关键用户旅程、团队约定、外部文档链接以及频繁被问及的问题的答案。你只需构建一次上下文，后续每个会话从此都在知情的基础上开始。不再有每次对话都要"这是我们产品的基本情况"的序言。

这就是智能助手和团队成员的区别。团队成员有上下文。拥有维护良好的 CLAUDE.md 的 Claude Code 也有上下文。

**子智能体生成（Subagent spawning）。** Claude Code 可以启动额外的 Claude 实例，并行或在特定专注领域处理子任务。作为 PM，你很少直接调用此功能，但它是复杂工作流的基础。

例如：你要求 Claude Code 准备一份竞争分析，将你的产品与三个竞争对手在功能、定价和定位方面进行比较。它不必按顺序执行，而是可以生成三个子智能体（每个负责一个竞争对手）并行研究，然后将结果汇总到比较矩阵中。这会花费你三个小时的标签页切换时间，缩短到 10 分钟。

这里的成本是真实的（你同时运行多个 Claude 实例），但对于高价值的研究任务，节省的时间往往证明这项投入是合理的。

**通过 MCP 连接外部系统。** 模型上下文协议（Model Context Protocol）让 Claude Code 可以连接到外部工具：Jira、Slack、Figma、数据库、分析平台。这是最新的能力，对于入门来说最不重要，但它消除了破坏工作动力的"导出-导入"循环。

你不必再导出 Jira 工单为 CSV、上传到 Claude.ai、再将结果复制粘贴回去，而是 Claude Code 直接查询 Jira、就地分析、并用发现结果更新工单。工作流保留在一个环境中。第 10 章将详细介绍，但目前只需知道，一旦掌握了基础知识，这是可行且强大的。

这五项能力有一个共同模式：它们消除了思考和行动之间的手动交接。Claude.ai 负责思考，你负责执行。Claude Code 负责思考并执行，然后向你展示结果以供批准。一次性问题两者皆可。可复现的 PM 工作流则受益于自主性。

**智能体循环（Agent Loop）** 是 Claude Code 实际运作的方式。你给它一个目标，它规划方法，执行一个步骤（读取文件、运行命令、生成产物），观察结果，决定继续迭代还是停止，然后向你报告。

**图 1.3：智能体循环** ——用于理解 Claude Code 如何运作。持续循环：接收目标 → 规划方法 → 执行步骤（读取文件、运行命令、生成产物）→ 观察结果 → 决定迭代或停止 → 报告。这个循环是 Claude Code 会话与对话感觉不同的原因——你在观看一个智能体解决问题。

这个循环是 Claude Code 会话与 Claude.ai 对话感觉不同的原因。你观看一个智能体解决问题，看到每一步，并对重要操作保持否决权。起初，详细程度可能感觉过度（"我现在要读取这个文件"），但这建立了信任。你在它提交更改之前看到它在做什么。

对于习惯了要么每件事都需要手动操作、要么像黑盒一样运行的工具的 PM 来说，这种中间地带需要适应。Claude Code 在做工作的同时展示它的工作过程。这是一种让你保持监督的协作模式。

Claude Code 是一个具有工作区访问权限和持久化能力的智能体。它与 Claude.ai 底层使用相同的模型，但能力不同：文件操作、shell 命令和上下文保持。这些能力解决了 PM 的特有问题，因此下一节提供了一个选择正确工具的决策框架。

## 1.3 决定何时使用哪个工具

在 Claude.ai 和 Claude Code 之间选择归结为三个问题：任务是否需要读写文件？你会再次执行这个任务吗？输出是否需要存放在版本控制中？如果任何一个答案是"是"，就从 Claude Code 开始。如果三个答案都是"否"，使用 Claude.ai。

大多数 PM 搞反了。他们默认使用 Claude.ai，因为它更熟悉，然后在需要引用文件或重复工作流时发现各种摩擦。从正确的工具开始可以省去后续的迁移成本。

**图 1.4：工具选择决策树** ——用于为每项任务选择正确的工具。三个问题：是否涉及文件？是否会再次执行？输出是否需要版本控制？任意"是"→ Claude Code。全部"否"→ Claude.ai。不确定时，默认使用 Claude Code 的只读模式。

以下是十二个常见 PM 场景的参考表。建议收藏此页。

| 任务 | 工具 | 原因 |
|------|------|------|
| 起草利益相关者邮件 | Claude.ai | 一次性对话任务，无需文件 |
| 通过代码库访问调查 bug 报告 | Claude Code | 需要阅读代码、git 历史、生成产物 |
| 头脑风暴功能创意 | Claude.ai | 探索性对话，无需持久化 |
| 分析客户反馈 CSV | Claude Code | 文件输入、结构化输出、大概率可重复 |
| 根据文档解释技术概念 | Claude.ai | 快速回答，无需文件操作 |
| 根据 git 历史生成发布说明 | Claude Code | 需要 git 命令、文件输出、每月可重复 |
| 基于研究笔记撰写 PRD 部分 | Claude.ai | 如果笔记在脑海里；如果在文件中则用 Claude Code |
| 了解功能 X 如何实现 | Claude Code | 需要代码库导航和上下文持久化 |
| 竞品功能对比 | Claude Code | 如果可重复；一次性则用 Claude.ai |
| 综合用户访谈记录 | Claude Code | 多文件输入、结构化输出产物 |
| 审查并建议 API 文档改进 | Claude Code | 需要读取仓库中的文档、建议文件修改 |
| 从零创建用户画像 | Claude.ai | 初稿是对话式的；如果数据驱动则用 Code |

模式是：**Claude.ai 用于思考，Claude Code 用于执行**。需要将输出复制到另一个工具的任务，意味着更适合 Claude Code。

**"我还会再做一次吗？"** 这个启发式问题是最快的决策规则。如果你每周都要做这个分析、每月都要创建这个报告、或反复调查这类问题，Claude Code 的设置成本立刻就能回收。即使第一次运行耗时更长（学习命令、结构化提示词、保存技能），第二次运行就快 10 倍。

PM 总是低估重复性。你以为自己只做一次竞争分析，但实际上你在建立一个季度审查流程。你以为自己只写一个用户故事，但实际上你在定义一个你接下来要用 50 次的模板。如果有任何可能你会重复这个任务，偏向 Claude Code，现在就建立可复用的工作流。

相反，如果这真的是一次性的（起草某条特定信息、探索一个模糊概念、获得快速答案），Claude.ai 更快。无需设置，无需学习曲线，只需提问然后继续。启动 Claude Code 进入目录、授权权限、结构化文件输出的开销，只有在你可以摊销这些成本时才有意义。

当你真正不确定时，默认使用 **Claude Code 的只读模式**。使用 `--permission-mode plan` 标志启动，这会阻止任何修改。提出你的问题。如果 Claude Code 的回答涉及"我会读取这些文件并生成这个产物"，那你就用对了工具。如果回答是纯粹的对话，没有文件引用，那你本可以用 Claude.ai。久而久之，这种校准会变成直觉。

一个边界情况：协作性工作。如果输出需要由工程团队审查、存放在代码仓库中或接入 CI/CD 流程，那么无论重复性如何，Claude Code 都是必需的。一份存放在 `docs/` 目录中、与所引用代码并列的调查报告，比 Slack 线程中的同样摘要更有价值。基于文件的产物持久存在、链接到具体提交、与团队工作流集成。而对话式输出会消失。

这个框架帮助你根据任务需求匹配工具能力。两个工具使用相同的模型，区别在于文件访问、命令执行和持久化。大多数 PM 工作涉及文件、迭代和协作。Claude.ai 优雅地处理 30% 的 PM 使用场景，而 Claude Code 处理另外 70% 能真正改变你工作方式的场景。

## 1.4 PM 必须使用 Claude Code 的三大场景

三类 PM 工作需要 Claude Code 的能力。如果你的任务属于其中任何一个，工具选择就已经决定了。

**图 1.5：PM 的三大使用场景** ——用于识别何时 Claude Code 必不可少。代码库调查（无需亲自阅读代码即可了解实现）、上下文产物生成（引用并与代码同步的文档）、以及可复现工作流（编码了重复性 PM 流程的技能）。如果你的工作涉及其中任何一个，Claude Code 就变成了必要的基础设施。

**代码库调查。** 你需要了解某事是如何工作的，但你无法流畅阅读代码，而工程团队有其他的优先事项。这是 Claude Code 对 PM 最经典的使用场景。

客户报告折扣码可以叠加使用，但本不应该。工程团队正全力投入季度目标。你需要判断：这是真正的 bug 还是用户操作失误？如果是真的，有多严重？大概范围是多大？

在代码仓库中打开 Claude Code。问："折扣码验证是如何工作的？请告诉我多个折扣码可能在哪些地方被应用于同一订单。"几分钟内，你就有了文件引用、用简明英文解释的函数，以及关于 bug 的假设。你整理出带有具体代码位置的调查报告。工程团队看到报告，确认问题，直接修复，跳过了通常的发现阶段。PM 总耗时：15 分钟。工程团队节省的时间：一小时的调查加上多次来回澄清。

这个模式不断重复。"认证是如何工作的？""支付失败时会发生什么？""应用在哪里检查订阅状态？"这些问题需要阅读代码、追踪数据流和理解实现细节。Claude Code 通过自然语言查询提供这些能力。

输出存放在 `docs/investigations/` 或相关工单旁。当类似问题出现时，未来的你会受益。未来的团队成员会从文档化的机构知识中受益。调查变成了一个产物，而不是一次丢失的对话。

**上下文产物生成。** 你需要创建一份引用代码库、引用外部研究或需要与代码变更保持同步的文档。文档的价值来自其与代码仓库的集成。

例子：一份引用当前 API 能力并链接到相关文件的 PRD。从 git 历史和 Jira 工单生成的发布说明，用你的产品语调撰写。竞争分析，每季度以相同结构更新，存放在 `research/competitors/` 目录下。随实现演进而保持更新的文档。

Claude.ai 可以写这些文档，但你手动提供上下文、手动保存输出、在事物变化时手动更新。Claude Code 从你的代码仓库读取，通过 MCP 从外部来源获取信息，在原地生成产物，并在你需要下一次更新时重新运行工作流。

第一份季度竞争分析作为技能设置需要 30 分钟。第二次运行只需 5 分钟。第三次、第四次、第五次每次只需 5 分钟。你已经建立了一个可复现的流程，产出结果一致，产物存放在版本控制中，团队可以看到其演变过程。

**可复现工作流。** 你定期做相同类型的任务：每周反馈综合、每月发布规划、季度路线图更新、每个功能的用户故事生成。具体输入会变，但流程是稳定的。

这就是技能（skill）发挥作用的地方。技能是存储在 `.claude/skills/` 中的文档化工作流，Claude Code 按需执行。你只需定义一次流程：所需输入、执行步骤、输出格式、质量标准。然后你调用它："对 `data/customer-feedback-nov-2024.csv` 运行 feedback-synthesis"，Claude Code 执行工作流。

没有技能，你每次都要重复相同的指令（解释分类方案、输出格式、综合深度）。有了技能，工作流被编码下来。你只需关注输入和审核输出。认知负担从"记住我上个月是怎么做的"降到"运行那个东西"。

第 8 章和第 9 章将深入介绍技能。目前，只需知道，如果你做的事超过两次，你就是构建技能的候选者。投入的努力在第三次运行时就能回本。

这三大使用场景（代码库调查、上下文产物生成和可复现工作流）涵盖了 PM 使用 Claude Code 的大部分高价值应用。如果你的工作不涉及这些，Claude.ai 可能就够了。如果你的工作涉及全部三项，Claude Code 就变成了必要的基础设施。

## 1.5 开始之前，了解权衡

Claude Code 解决了真实的问题，但它也带来了 Claude.ai 可以避免的成本和限制。

**学习曲线。** 你需要安装它、理解权限模式、学习基本命令、并培养何时使用哪种能力的直觉。这需要数小时的动手实践。网页界面无需安装，立即可以使用。对于偶尔需要 AI 帮助但很少与代码仓库打交道的 PM 来说，切换成本可能超过收益。

预计第一周会感觉效率低下。你会对命令感到笨拙，忘记在想要只读时设置只读模式，并怀疑这是否值得。到了第二周，如果你在正确的任务上使用它，效率提升开始显现。到了第四周，你会讨厌不得不使用 Claude.ai，因为相比之下它感觉功能受限。但最初几次会话确实会让人感到别扭。

**成本可见性和预算。** Claude.ai 采用订阅定价：固定月费，在速率限制内无限制使用。Claude Code 使用 API 定价（按消耗的 token 付费）。对于探索性较强的 PM 工作，token 消耗可能难以预测。阅读大型代码库、生成多个子智能体或运行复杂的综合任务，可能以订阅定价掩盖的方式消耗你的月度预算。

第 2 章详细介绍了成本管理。简要版：典型 PM 用量预算 $50-150/月，每次会话后用 `/cost` 监控，当成本攀升时使用 `/compact` 减少上下文。如果你对变动成本感到不安或缺乏预算审批权限，Claude.ai 的固定定价可能更可取，尽管有局限性。

**企业部署。** 许多组织限制 API 访问，要求对具有文件系统访问权限的工具进行安全审查，或禁止连接到外部服务。Claude Code 需要 Anthropic API 密钥，并以本地文件权限运行。如果你的组织的安全策略禁止这样做，你只能使用他们批准的官方工具（可能只有网页界面）。

这不是 Claude Code 的限制，而是企业 IT 治理的现实。为获批使用而倡导可以奏效，特别是如果你先在个人项目上展示价值，但要预期一个数周的审批流程。

**Claude.ai 确实更好的场景。** 快速提问。头脑风暴会议。粘贴内容并希望获得对话式反馈的草稿审查。任何你还未定义输出或不需要文件操作的探索性工作。网页界面打开更快，无需考虑权限的心理负担，在任何设备上无需安装即可使用。

如果你发现自己在用 Claude Code 做更适合网页界面的任务，那你正在优化错误的事情。两种工具的存在是因为它们服务于不同的需求。为每个任务使用合适的工具，而不是把一切都强加在你偏好的界面上。

## 1.6 双工具心智模型

两种工具，相同的智能，不同的能力。根据任务需求选择，而不是哪个界面感觉更熟悉。

**Claude.ai：对话式助手。** 你说，它回应。每次对话从零开始。输出留在对话历史中，除非你手动复制到别处。理想用于思考问题、获取快速答案、起草不需要与文件或代码仓库集成的内容。零设置成本，零学习曲线，固定订阅定价。

**Claude Code：具有工作区访问权限的自主智能体。** 你设定一个目标；它读取文件、运行命令、生成产物，并报告结果。对话建立在来自 `CLAUDE.md` 的持久项目上下文之上。输出落在你的文件系统中，受版本控制，对团队可见。理想用于代码库调查、产物生成和可复现工作流。需要安装和学习，变动 API 成本，更高的认知负担。

**决策框架：**

- 这个任务是否涉及文件？→ Claude Code
- 我会再次做这个任务吗？→ Claude Code
- 输出需要存放在代码仓库中吗？→ Claude Code
- 这是一次性对话任务吗？→ Claude.ai
- 我在做没有明确输出的头脑风暴吗？→ Claude.ai

大多数 PM 会经常使用两种工具。Claude.ai 用于 30% 的任务，Claude Code 用于 70%。错误在于对所有人都用 Claude.ai，因为它感觉更容易，然后困惑为什么 AI 并没有真正改变你的工作流程。那些会被改变的工作流，是产物重要的、重复性重要的、与代码仓库集成重要的工作流。

Claude Code 处理这些。这就是本书存在的原因：向你展示如何识别这些工作流、正确地构建它们、并随着时间复利价值。第 2 章介绍设置和安全性。让我们开始吧。
# 第 2 章 安装、安全模式与成本管理

## 2.1 15 分钟内让 Claude Code 运行起来

你已经决定 Claude Code 能解决你的问题。现在你需要让它跑起来而不搞砸任何事情。本节假设你有一个终端并能执行命令，在 15 分钟内带你从零到可以工作的会话。

安装使用平台特定的命令。选择与你操作系统匹配的命令：

**macOS、Linux 或 WSL：**
```
curl -fsSL https://claude.ai/install.sh | bash
```

**Windows（PowerShell）：**
```
irm https://claude.ai/install.ps1 | iex
```

**macOS 使用 Homebrew：**
```
brew install --cask claude-code
```

**任何平台（需 Node.js 18+）：**
```
npm install -g @anthropic-ai/claude-code
```

推荐使用 `curl`/`irm` 方法。它下载一个原生二进制文件，可以直接运行。npm 选项需要 Node.js 18 或更高版本，耗时更长，但跨平台通用。如果你的组织限制软件安装，你需要先获得 IT 批准。

安装大约需要两分钟。如果在 macOS 或 Linux 上使用 curl 方法看到权限错误，脚本会提示你输入密码。Windows PowerShell 可能需要以管理员身份运行。

**认证**在安装后于 Claude Code 内处理。你有两个选项：

**选项 1：Claude 订阅（Pro、Max 或 Team 计划）。** 启动 Claude Code 后，系统会提示你认证，或者你也可以使用：
```
/login
```
这个斜杠命令会打开浏览器让你用 Claude 账户凭据进行认证。使用量包含在你的订阅中，没有按 token 计费，没有意外收费。对大多数 PM 来说这是更简单的选项。

**选项 2：Anthropic API（按用量计费）。** 如果你没有 Claude 订阅，从 `console.anthropic.com/settings/keys` 获取 API 密钥。创建一个密钥，立即复制（你之后看不到它了），然后将其设置为环境变量。有关永久设置环境变量的方法，请参考你的操作系统文档。

> **关键警告**：如果你设置了 `ANTHROPIC_API_KEY` 环境变量，Claude Code 会自动使用它进行 API 计费（按用量付费）并忽略你的订阅，即使你有 Pro 或 Max 计划。你将按 API 费率计费，而不是使用你的订阅。如果你使用订阅，请确保此环境变量未被设置。使用 `/status` 验证当前激活的认证方法。

认证信息存储在本地主目录的配置中。绝不要将 API 密钥提交到版本控制或与团队成员共享。每个人需要自己的认证信息，以便追踪使用情况和明确计费。

**验证安装**：导航到任意目录并运行：
```
claude doctor
```
这会检查你的安装，显示版本，并确认一切配置正确。如果你看到 "command not found"，说明安装失败。尝试重新安装或检查安装目录是否在你的 PATH 中。

**常见安装故障：**

- **网络限制。** 企业防火墙有时会阻止来自 claude.ai 或 npm 注册表的下载。如果安装挂起或因网络错误失败，你可能需要配置组织的代理或请求例外。如果你提供安装 URL，IT 通常可以在一天内解决。
- **权限拒绝。** 在 macOS 和 Linux 上，curl 安装脚本可能需要提升权限才能在系统目录中安装二进制文件。脚本会提示你输入密码。如果使用 npm，你可能需要 `sudo npm install -g @anthropic-ai/claude-code` 进行全局安装。
- **Windows 执行策略。** PowerShell 可能因执行策略限制阻止安装脚本。如果你看到此错误，在安装命令前运行 `Set-ExecutionPolicy RemoteSigned -Scope CurrentUser`，或请 IT 调整策略。
- **Node.js 版本不匹配。** 如果使用 npm 安装方法，Claude Code 需要 Node.js 18 或更高版本。用 `node --version` 检查。如果你的版本较旧，请先使用 nvm（macOS/Linux）或 nvm-windows（Windows）等版本管理器更新 Node.js。
- **认证失败。** 如果 `/login` 打开了浏览器但认证失败，检查你是否拥有有效的 Claude Pro、Max 或 Team 订阅。如果使用 API 密钥，验证你复制了完整的密钥，没有多余的空格或换行符。必要时重新生成，因为旧密钥不会自动过期。

**配置**使用三级层次化设置系统：

- **全局**：`~/.claude/settings.json`（适用于所有项目）
- **项目**：`.claude/settings.json`（团队共享，提交到 git）
- **本地**：`.claude/settings.local.json`（个人覆盖，被 gitignore）

刚开始时你很少需要直接编辑这些文件，但了解这个结构有助于后续使用。这些文件存储认证信息、默认权限模式和其他偏好设置。

要为 PM 设置推荐的默认权限模式，创建 `~/.claude/settings.json`，内容如下：

```json
{
  "defaultMode": "plan"
}
```

这使每个会话都以只读模式启动，直到你主动覆盖，这是学习期间的安全默认设置。第 2.3 节解释了为什么这很重要。

安装完成。你已经安装、认证并配置了 Claude Code。如果你遇到超出这些常见故障的问题，请查看 `code.claude.com/docs` 的故障排除指南。大多数 PM 可以顺利完成安装。时间消耗通常在于等待 IT 批准，而不是技术困难。

## 2.2 运行你的第一次调查

打开终端，导航到你在使用的任意 git 仓库。这可以是你的产品主仓库、文档仓库，甚至是个人项目。Claude Code 可以在任何包含文件的目录中工作，但在你需要理解的仓库中最有用。

```bash
cd /path/to/your/repository
claude
```

会话启动。你会看到欢迎信息、关于 Claude Code 可以访问什么（当前目录及其子目录）的说明，以及等待输入的提示符。

会话界面刻意保持简洁。你有三个要素：

1. **提示符**：一个 `>` 符号，你可以在这里用自然语言输入请求。不需要特殊语法。就像在给一个能读文件、运行命令的同事发消息一样书写。
2. **输出和状态指示器**：Claude Code 向你展示它正在做什么。"正在读取文件 X。""正在运行命令 Y。""正在生成产物 Z。"起初这种详细程度可能感觉过度（你知道自己问了什么），但这建立了信任。你在操作发生之前看到每个动作，当操作涉及修改文件时这一点很重要。
3. **Token 用量指示器**：每次回复后，你会看到消耗的 token 和大致成本。这是你的预算感知反馈循环。第一次会话时可以忽略它；到第三次会话时开始关注，届时你会理解什么驱动了成本。

**理解 Claude Code 启动时看到什么**：你当前位置的每个文件和子目录。如果你在仓库中，还有 git 历史。任何用于持久化上下文的 `CLAUDE.md` 文件。它被允许访问的环境变量。它不会自动读取每个文件（那会消耗大量 token），但可以读取你引用的或它判断与你请求相关的任何文件。

试试这第一个请求：

> **这个仓库是什么？用三句话概括它的功能。**

Claude Code 会读取关键文件如 `README.md`、`package.json` 或类似的元数据，然后提供摘要。回复会显示它查阅了哪些文件。这教会你模式：你提问，它决定读取什么，向你展示它读了什么，然后回答。

**你立即需要的基本命令：**

- `/help`：显示可用命令。你不会记住这些，但在第一周你会经常参考这个列表。
- `/status`：显示当前会话状态，包括权限模式、已修改的文件、消耗的 token 和估算成本。
- `/exit`：结束会话。你的对话历史被本地保存以供参考，但上下文不会持久化到下一个会话，除非你创建了 `CLAUDE.md` 文件。

试试第二个请求：

> **显示这个仓库中最近修改的五个文件。**

Claude Code 运行 git 或文件系统命令，显示结果，并解释发现的内容。你正在看到 shell 命令访问的实际运作：你通过自然语言获得的能力，无需知道确切的命令语法。

**你的第一次会话应该持续大约五分钟。** 提出两到三个关于你所在仓库的探索性问题。用 `/status` 检查 token 消耗。用 `/exit` 退出。就这样。你已经确认安装有效并体验了核心交互模式。

第二次会话会感觉不那么陌生。到第三次会话，你将不再关注界面，而是专注于工作。基本使用的学习曲线以小时计，而不是天，前提是你正在使用你已经从产品层面理解的仓库。

## 2.3 控制 Claude Code 能做什么

Claude Code 可以修改文件、运行任意 shell 命令，如果你要求，还可以删除东西。在理解自己在做什么之前，这种能力需要防护栏。

**四种权限模式**控制 Claude Code 可以自主完成的事情。作为 PM，你主要使用其中三种。

权限模式梯度（从最严格到最不严格）：

- **plan 模式（只读分析）**：Claude Code 可以读取文件，运行只读命令如 `git log` 或 `ls`，并回答问题。它不能修改文件、安装依赖、运行测试或执行改变状态的命令。在你有具体的修改需求之前，这是你作为 PM 的推荐默认模式。

  **适用场景**：调查代码库、分析数据、理解实现、分类 bug。90% 的 PM 工作属于此类别。

- **acceptEdits 模式（自动批准文件修改）**：Claude Code 可以读取和修改文件，无需每次编辑都提示。它仍会在运行 shell 命令前提示。当你信任 Claude Code 进行文件更改但希望在命令执行上有监督时，此模式很高效。

  **适用场景**：生成发布说明、创建 PRD 文档、更新 CLAUDE.md、构建技能。需要文件输出但不运行构建或测试的任务。

- **default 模式（对所有操作提示）**：Claude Code 在修改文件或运行命令前请求权限。这提供了最大的控制（你批准每个操作），代价是更多的打断。

  **适用场景**：学习 Claude Code 并希望在操作发生前看到每个动作，或处理敏感更改时需要明确批准。

- **bypassPermissions 模式（完全自主）**：Claude Code 可以在不提示的情况下执行你在终端中能做的一切。运行测试、安装包、启动服务器、进行 git 提交、推送到远程仓库。此模式功能强大但也充满风险。

  **适用场景**：作为 PM 很少使用。主要用于与工程师一起测试某个东西或调试需要完整系统访问权限且你完全信任 Claude Code 的问题。大多数 PM 工作流永远不需要此模式。此模式需要在启动时使用 `--dangerously-skip-permissions` 标志。它被刻意设计为难以启用。

**设置权限模式有几种方式：**

1. `settings.json` 中的默认值（第 2.1 节）适用于所有会话
2. 会话期间按 `Shift+Tab` 在三种常见模式间循环：default → acceptEdits → plan
3. 启动时的命令行标志：`claude --permission-mode plan`

进入会话后，`Shift+Tab` 循环最快。如果你以 default 模式启动，然后意识到你只是想调查而没有修改风险，按两次 `Shift+Tab` 循环到 plan 模式。注意，bypassPermissions 模式不能通过 `Shift+Tab` 启用。你必须在启动时使用 `--dangerously-skip-permissions` 标志，这是一种安全机制。

**plan 模式用于安全探索是你的训练轮。** 如果你将其配置为默认（第 2.1 节），每个会话都安全地开始。你可以随时按 `Shift+Tab` 切换到更宽松的模式，如果发现需要文件修改能力的话。文件修改或命令执行一旦完成，你无法撤销。

**具体场景**：你正在调查一个 bug，要求 Claude Code 展示认证机制如何工作。在 plan 模式下，它读取相关文件并进行解释。完美——你获得了需要的信息。在 bypassPermissions 模式下，如果你的请求措辞模糊，Claude Code 可能尝试运行认证流程，可能触及外部服务或修改数据。这在请求明确时不太可能发生，但 plan 模式完全消除了这种可能性。

**PM 何时使用每种模式：**

| 任务 | 模式 | 原因 |
|------|------|------|
| 了解功能 X 如何工作 | plan | 只读调查 |
| 分类 bug 报告 | plan | 仅读取代码和 git 历史 |
| 生成发布说明文档 | acceptEdits | 需要写入文件输出 |
| 在 .claude/skills/ 中创建新技能 | acceptEdits | 写入技能配置文件 |
| 分析客户反馈 CSV | plan | 读取数据、回答问题、无需文件输出 |
| 用新上下文更新 CLAUDE.md | acceptEdits | 修改仓库文件 |
| 运行测试复现 bug | default 或 acceptEdits | 命令执行需要批准 |
| 调查数据库模式 | plan | 只读，除非需要运行迁移 |

第 2.1 节已介绍在配置中设置默认模式，但值得重申：在 `~/.claude/settings.json` 中添加 `"defaultMode": "plan"` 使每个会话安全地启动。当你需要文件修改或命令执行时，随时可以按 `Shift+Tab` 切换到更宽松的模式。默认为最严格的模式可以防止事故。

**"事故"是什么样子**：你在 bypassPermissions 模式下要求 Claude Code "修复 README 中的拼写错误"。它读取 `README.md`，识别拼写错误，修正它，并且（因为你没有另外说明）以自动生成的消息进行 git 提交并推送到你的当前分支。这可能是有帮助的，也可能违反你团队的提交规范，并需要用 force-push 来撤销。在 acceptEdits 模式下，它只会修复文件中的拼写错误然后停止，让你在提交前进行审查。

这些模式的存在是因为 Claude Code 足够强大，如果使用不慎会导致问题。作为 PM，你很少会在接近它能力极限的地方操作，所以严格的默认设置在学习期间保护你的安全。

澄清一点："安全"意味着"不会意外修改东西"。Claude Code 在任何模式下都会根据读取的内容准确回答问题，但不能保证其理解是正确的。如果你问"认证是如何工作的？"，它误解了代码流程，给你错误的信息——这不是安全问题，而是准确性问题。权限模式防止非预期的操作，而不是错误答案。第 3.6 节讨论代码库调查的限制。

## 2.4 理解你在为什么付费

Claude Code 每次使用都会产生费用，但付款方式取决于你的认证方法。如果你使用 Claude 订阅（Pro、Max 或 Team），使用量已包含在内，可以忽略本节的大部分内容。如果你使用 API 按用量计费，请仔细阅读。

**对于 Claude 订阅用户**：你的使用量已捆绑到月度订阅费中。没有按 token 收费，没有意外账单，无需过度追踪成本。`/cost` 命令（第 2.5 节）仍会显示 token 消耗供你了解，但不会转化为额外费用。可跳至第 2.5 节。

**对于 API 用户**：Claude 读取或写入的所有内容都计入你的 token 预算。输入 token 是你发送的文本（你的提示词以及 Claude 为回答而读取的任何文件）。输出 token 是 Claude 生成的文本（回复、写入的文件内容、解释）。Anthropic 的 API 定价大约是每百万输入 token 3 美元，每百万输出 token 15 美元（Claude Sonnet，标准模型）。Opus 成本更高，Haiku 成本更低，但 Sonnet 是 Claude Code 的默认模型，能很好地处理 PM 任务。

**将 token 转化为实际使用量**：一个 token 大约是四分之三个单词。一篇 500 字的文档大约是 650 个 token。一个典型的代码库文件根据长度可能是 200-2,000 个 token。你要求 Claude Code 调查某个东西的提示词是 50-200 个 token。回复是 200-1,000 个 token。

算一下：要求 Claude Code 读取五个文件（5,000 token 输入）并提供摘要（500 token 输出）大约花费 $0.02。要求它读取五十个文件并生成一份综合报告可能花费 $0.50。一次深度调查会话，你跨数十个文件迭代多次，可能消耗 200,000 token，花费 $1-2。

**上下文窗口：它是什么，为什么重要**。每个活跃的 Claude Code 会话维护一个上下文窗口：你对话的运行历史、所有已读取的文件和所有已生成的输出。这个上下文是 Claude Code "记住"你之前讨论内容的方式。上下文窗口有大小限制（出版时 Claude Sonnet 为 200,000 token，但这个限制会定期增加）。

上下文窗口在会话期间增长。每次交换都会增加上下文：初始提示词（小）、文件读取（中）、回复（中）、后续问题（累积）。到第 10 轮交换，上下文已经显著增长。`/compact` 的效果是将累积的上下文压缩为摘要。

对 PM 来说，这体现为两个实际的限制：

1. **长会话会变得昂贵。** 你发送的每条消息和 Claude 生成的每个回复都保留在上下文中。经过二十轮交换，你可能拥有 50,000 token 的上下文。每个新提示词在回复前都会读取整个上下文，所以你每次请求都在为重新处理之前的对话历史付费。这会悄然累积。
2. **会话可能超出上下文窗口。** 如果你调查大型代码库、要求 Claude Code 读取许多文件并广泛迭代，你会触及上下文限制。Claude Code 会警告你，可以使用 `/compact` 总结和减少上下文，但要注意，非常长的会话需要管理。

**每种任务类型的近似成本：**

| 任务 | Token | 成本 | 耗时 |
|------|-------|------|------|
| 关于文件的快速问题 | 2,000-5,000 | $0.01-$0.02 | 1 分钟 |
| 功能调查（X 如何工作？） | 20,000-50,000 | $0.10-$0.25 | 5-10 分钟 |
| 带代码库分析的 bug 分类 | 30,000-100,000 | $0.15-$0.50 | 10-20 分钟 |
| 从 git 历史生成发布说明 | 40,000-80,000 | $0.20-$0.40 | 5-10 分钟 |
| 综合客户反馈 CSV | 50,000-150,000 | $0.25-$0.75 | 10-15 分钟 |
| 通过迭代构建新技能 | 60,000-120,000 | $0.30-$0.60 | 15-30 分钟 |
| 深度代码库探索会话 | 100,000-300,000 | $0.50-$1.50 | 30-60 分钟 |

这些是估计值。你的实际消耗取决于代码库大小、问题复杂度和迭代深度。在最初几次会话后使用 `/cost`（第 2.5 节）来校准你的直觉。

**为什么 PM 消耗 token 比工程师快。** 工程师知道他们在找什么。他们提出精准的问题："显示 `user.authenticate()` 的实现。"他们得到答案，据此行动，然后退出。总 token：10,000。成本：$0.05。

PM 是在探索。你不知道哪个文件包含相关逻辑。你提出宽泛的问题："这个应用中的认证是如何工作的？"Claude Code 读取多个文件来构建完整图景。你追问以澄清边缘情况。你迭代解释直到理解。总 token：80,000。成本：$0.40。

这不是低效。这是 PM 调查的本质。你从零开始构建理解，而工程师是在增强已有知识。要为探索成本做预算。替代方案是反复打断工程师，这在团队生产力方面有其自身的成本。

**典型 PM 使用模式的月度预算（仅 API 用户）**：

Claude Code 用户的真实数据展示典型支出：
- 平均使用量：~$100-200/月（日常活跃用户）
- 轻度使用（5-10 次/月，主要是快速问题）：$10-30
- 中度使用（15-25 次/月，调查和生成混合）：$50-100
- 重度使用（每日会话，深度调查，技能构建）：$100-200
- 超级使用（每日多次会话，大量代码库工作）：$200-400

如果你作为 PM 每月花费超过 $400，你要么在一个巨大的代码库上工作，要么在跑不必要的超长会话，要么在用 Claude Code 做更适合其他工具的任务。第 2.5 节介绍成本管理策略。

在你看到 Anthropic 账单之前，成本感觉是抽象的。先从预算 $100/月开始，通过 `/cost` 监控实际使用量。根据交付的价值调整。如果 Claude Code 一个月为你节省五小时的工作，$100 相比于你每小时的全额成本只是一个舍入误差。如果你花了 $100 却得到极少的价值，那你是用错了，可能在用它做 Claude.ai 处理得更好的任务。

**对于订阅用户**：你的成本固定在订阅费率（出版时 Pro $20/月，Max $40/月）。Token 消耗有助于理解会话效率，但不影响你的账单。

Token 意识很重要，无论你是按 token 付费还是使用订阅。订阅用户仍然受益于高效的会话：更短的对话意味着更快的答案和更少的上下文需要管理。API 用户有直接成本控制的额外激励。无论哪种方式，你都会学会提出更好的问题，并在获得所需信息后退出会话，而不是在递减回报的对话中徘徊。

## 2.5 追踪和控制支出

Claude Code 为你提供三个命令来在会话中监控和控制支出。习惯性地使用它们，直到成本意识变得自动化。

**`/cost`：查看当前会话支出。** 随时运行此命令查看消耗的 token（输入和输出分开）、当前会话成本估计以及上下文窗口使用情况。这是你的预算仪表盘。

对于订阅用户，显示的金额是按 API 费率估算的会话成本。这对理解会话效率很有用，即使你不按 token 付费。对于 API 用户，这些是实际费用。

在每个会话的前几个问题后使用 `/cost` 建立基线。在提出你知道会很昂贵的问题之前再次使用，比如为进行全面分析而读取五十个文件。在会话结束时使用，以建立关于不同类型工作成本的直觉。

输出显示 token 计数和估计成本：
```
Session costs:
Input tokens:  45,230 ($0.14)
Output tokens: 12,100 ($0.18)
Total:         $0.32
```

对于 API 用户：这确切地告诉你已经花了多少钱。如果你进入会话十分钟就已经花到 $2，有什么不对劲。要么你正在调查一个庞大的代码库，要么迭代过度，要么提出的问题需要读取远超必要的上下文。调整你的方法，或者如果价值合理则接受成本。

对于订阅用户：这告诉你会话效率。高 token 计数意味着更长的会话和更多需要管理的上下文，这会影响响应速度，即使不影响你的账单。

**`/compact`：压缩上下文以减少 token。** 当会话成本攀升（或上下文变得难以管理）而你仍需继续工作时，`/compact` 触发 Claude Code 总结迄今为止的对话历史和文件内容，用压缩版本替换完整上下文。这减小了上下文窗口大小，降低了后续消息的 token 成本。

你可以选择性地指定保留什么：`/compact Focus on preserving authentication implementation details` 告诉 Claude Code 摘要中最重要的是什么。

具体来说：你花了 30 分钟调查一个复杂功能，累积了 80,000 token 的上下文。现在每个新问题都要处理整个上下文。对于 API 用户，即使简单请求也可能花费 $0.25。对于订阅用户，意味着更慢的响应。运行 `/compact`，Claude Code 将上下文压缩到 20,000 token，保留关键信息并丢弃冗余。后续问题更快更便宜。

**何时使用 `/compact`：**
- 会话成本超出任务预算
- 你在会话中切换话题，不需要之前的上下文
- 你想继续工作但减少每条消息的成本
- `/cost` 显示上下文大小接近窗口限制

**权衡**：压缩会丢失细节。如果你压缩了大量的调查笔记，然后提出一个需要那些细节的问题，Claude Code 将没有那些信息。你可能需要重新读取文件，再次消耗 token。在你完成一个工作阶段并转移到另一个阶段时使用 `/compact`，而不是在迭代调查中途。

**`/clear`：重新开始而不丢失已学内容。** 此命令结束当前会话并在同一目录中启动一个新会话，不携带上一个对话的任何上下文。与 `/exit` 后重新启动不同，`/clear` 更快，并保持你处于相同的工作状态。

**成本命令决策流程图**：从 `/cost` 检查会话状态开始。如果上下文可管理且成本可接受，继续工作。如果成本高但需要继续同一话题，使用 `/compact` 压缩。如果切换到不相关的任务或上下文过于膨胀，使用 `/clear` 重新开始。权衡：`/compact` 保留关键上下文但丢失细节；`/clear` 丢失一切但给你一张白纸。

**何时使用 `/clear`：**
- 你完成了一项任务，想开始另一项不相关的任务
- 会话成本高且 `/compact` 不够
- 你走进了一条无产出的路径，想要一张白纸
- 之前讨论的上下文正在混淆当前的回复

**权衡**：你丢失所有对话历史。如果你没有将输出保存到文件，那些输出就丢失了。在运行 `/clear` 之前，确保会话中任何有价值的信息都已捕获到文件或笔记中。

**设置支出提醒和限制（API 用户）**：Claude Code 没有内置的支出提醒，但你可以通过 `console.anthropic.com` 上的 Anthropic 账户仪表盘监控使用量。在那里设置计费提醒，当月支出超过你定义的阈值时通知你：$100、$200、$300，适合你的预算即可。

第一个月每周检查仪表盘以校准预期。大多数 PM 会惊讶于在调查大型代码库或频繁会话时使用量累积的速度。这是正常的。目标是有意识地在高价值工作上支出，而不是以牺牲生产力为代价最小化成本。

**月度预算工作流（API 用户）：**

1. 根据预期使用量决定月度预算（第 2.4 节指南）
2. 在 Anthropic 仪表盘中设置预算 80% 的计费提醒
3. 每次会话后使用 `/cost` 追踪每日消耗
4. 第一个月每周审查实际 vs. 预算
5. 根据交付的价值调整预算或使用模式

如果你总是在月中达到预算限制，要么增加预算（如果价值合理的话），要么减少会话频率和范围。如果你只花了预算的 30%，你要么过于保守，要么高估了自己的需求。考虑更积极地使用 Claude Code。

**订阅用户**：你的成本是固定的。使用 `/cost` 理解会话效率，但不要担心显示的金额。它们是估计值，不是实际收费。

**成本管理关乎有意识的选择，而非最小化。** 对 API 用户来说，每月花 $200 在一个为你节省 10 小时工作的工具上是惊人的投资回报率。对订阅用户来说，你的 $20-40/月已经花出去了，所以积极地使用 Claude Code 从中获取价值。使用这些命令来保持意识，就会话范围做出知情决定，并（对 API 用户而言）避免账单冲击。第一个月之后，成本管理将变成自动化的。你将凭直觉知道哪些任务消耗 token，以及效率是否证明了方法的合理性。

# 第 3 章 理解你的产品实现

## 3.1 PM 的代码库困境

你负责产品，但无法阅读代码。这形成了一个拖慢一切的依赖循环。客户报告你的定价计算器行为令人困惑。工程团队正在 sprint 中途。你需要答案：这是 bug 还是设计如此？如果是 bug，有多严重？可能的修复复杂度如何？你无法自己回答这些问题，于是你打断工程团队。他们切换上下文、调查、反馈。他们花两小时，你等两天。bug 停留在待分类状态。

工程团队不应该成为你接触自己所负责代码库的唯一接口。你不需要流畅阅读代码。你需要的是提出问题并获得准确答案，而不让某人的 sprint 偏离轨道。这就是 Claude Code 所提供的。

工程团队上下文切换的延迟成本非常昂贵，却不出现在 sprint 看板上。每个"快速问题"算上上下文切换，就是 15 分钟的中断。每天五个问题，就是一小时的工程时间加上你自己的等待时间。乘以一个季度，你就在本可以自己回答的问题上烧掉了数周的生产力。

**图 3.1：调查工作流对比** ——用于理解直接代码库调查相比传统工程依赖循环节省的时间。左：旧工作流，耗时两天，多次上下文切换。右：PM 通过 Claude Code 直接调查，10-20 分钟。

替代方案不是学会写代码，而是通过一个能将实现翻译成产品语言的智能体（agent）直接询问代码库。你仍然将复杂问题升级给工程团队，但你自己处理第一轮调查。本章将向你展示如何做到。

## 3.2 构建代码库的心智地图

在调查具体功能之前，花 20 分钟了解代码库的组织方式。这是你的地图。没有它，每次调查都从零开始。有了它，你知道该看哪里、该问什么问题。

**在第一次使用 Claude Code 进入新仓库时运行一次此会话。** 每季度或在重大架构变更落地时更新。输出成为一份参考文档，为后续每次调查节省时间。

在仓库根目录以 plan 模式启动 Claude Code：

```bash
cd /path/to/your/repository
claude --permission-mode plan
```

Plan 模式在这里至关重要（你是在学习，不是修改）。使用此提示词：

> **提示词：架构概览**
>
> 向产品经理解释这个代码库的架构。涵盖：
> - 主要组件有哪些，各自负责什么
> - 数据如何在系统中流转
> - 面向用户的功能通常在哪里
> - 前端和后端如何连接（如适用）
> - 我应该了解的关键抽象或模式
>
> 保持解释非技术化。聚焦于能帮助我在调查功能时导航代码库的心智模型。

Claude Code 会读取关键文件（通常是 `README.md`、包配置、目录结构和核心模块的抽样），然后提供摘要。对典型仓库，回复需要 5-10 分钟，成本 $0.10-0.25。

**输出中可预期的内容：**

- **组件分解。**"前端是 `src/client/` 中的 React 应用，处理所有 UI。后端是 `src/server/` 中的 Express API，管理业务逻辑和数据库访问。`src/shared/` 目录包含两者共享的代码。"
- **数据流解释。**"当用户提交表单时，React 组件向 `/api/endpoint` 发送请求，路由到 `src/server/controllers/` 中的控制器，调用 `src/server/services/` 中的服务执行业务逻辑，通过 `src/server/models/` 中的模型查询数据库。"
- **功能位置模式。**"面向用户的功能通常实现为 `src/client/components/` 中的 React 组件，对应的 API 端点位于 `src/server/routes/`，业务逻辑在 `src/server/services/` 中。"
- **关键抽象。**"代码库使用仓储模式（repository pattern）：数据库访问通过 `src/server/repositories/` 中的仓储类进行，而非直接查询。认证使用 JWT token，由 `src/server/middleware/auth.js` 中的中间件验证。"

这不是全面的文档，而是一个起始定向。你正在建立关于东西在哪里以及它们如何连接的直觉。当你以后调查"结账如何工作"时，你会知道去查看客户端组件、服务端路由和支付服务集成。

**将输出保存以备将来参考。** 将回复复制到 `docs/architecture-overview.md` 之类的文件中，或添加到你的 `CLAUDE.md`（第 3.5 节）。这成为你的快速参考指南。当三个月后需要刷新记忆时，你阅读文档而不是重新运行会话。

**有价值的后续问题**：在初始概览之后，针对与你的产品领域相关的方向提出澄清问题：
- 用户认证在哪里发生？我们如何确定已登录用户可以访问什么？
- 我们如何处理支付？我们集成了哪些第三方服务？
- 定价逻辑在哪里实现？是前端、后端还是两者？

这些问题花费另外 5-10 分钟和 $0.10-0.20，但它们填充了产品特定的细节。通用架构不如知道关键业务逻辑在哪里重要。

**何时此会话不够充分**：小型仓库（50 个文件以下）结构清晰，不需要这样。你可以在调查过程中学习布局。大型单体应用（数千个文件）需要多次聚焦会话，而不是一次概览。你需要分别理解特定子系统。

这次对齐会话是你的基础。当你第一次调查功能时，已经知道应该聚焦哪些目录，它立刻就能回本。工程师认为这些知识理所当然，因为他们每天都在代码库中工作。你正在构建同样的心智地图，只是通过对话而非阅读代码。

## 3.3 四个几乎能回答一切的问题模式

大多数 PM 关于代码库的问题属于四种模式。学会这些模式，你几乎可以无需工程帮助调查任何东西。

**图 3.2：四种调查模式** ——用于结构化你的代码库查询。这个 2x2 网格涵盖："X 如何工作？"（功能实现）、"X 访问什么数据？"（数据依赖）、"当 X 时会发生什么？"（边缘情况）、以及"为什么会出现 X？"（行为解释）。这四种模式处理 90% 的 PM 调查。

### 模式 1："[功能] 是如何工作的？"

你需要理解实现以回答产品问题。客户问为什么折扣码不能与促销叠加使用。工程团队构建了这个，但你不知道设计推理。与其让工程团队解释，不如询问代码库。

以 plan 模式启动 Claude Code，使用此提示词：

> **提示词：功能实现**
>
> 折扣码验证在这个应用中是如何工作的？具体来说：
> - 折扣码逻辑在哪里实现？
> - 什么规则决定折扣码是否可以应用？
> - 折扣码能否与其他促销叠加？原因是什么？
> - 当用户输入无效代码时会发生什么？
>
> 请用产品语言解释，而不是代码细节。

Claude Code 从用户操作（输入代码）追踪功能，经过验证逻辑，到结果（应用折扣或显示错误）。你获得文件引用和简明英文解释。"折扣验证发生在 `src/services/discount-validator.js:45-78`。系统检查代码是否活跃、未过期、满足最低购物车金额要求。第 92 行的业务规则阻止与促销叠加，当 `cart.hasActivePromotion` 为 true 时拒绝应用折扣码。"

这给了你答案（因为明确的业务逻辑而不能叠加）以及向客户或利益相关者解释的上下文。它还告诉你在需求变化时该看哪里。你现在知道哪个文件拥有折扣逻辑。

### 模式 2："[功能] 访问什么数据？"

理解数据依赖性有助于你评估变更影响，识别隐私或性能问题。你正在规划一个需要用户购买历史的功能。什么数据可用？数据从哪里来？隐私影响是什么？

> **提示词：数据访问调查**
>
> 购买历史功能访问什么用户数据？请展示：
> - 读取了哪些具体数据字段（订单详情、支付信息、收货地址等）
> - 这些数据存储在哪里（数据库表、外部 API、本地存储）
> - 代码中谁可以访问这些数据（哪些服务或组件）
> - 我应该了解的任何隐私或安全考虑

你了解到购买历史从 orders 表拉取数据（订单 ID、日期、商品、总额），与 products 表关联以获取当前产品名称，并出于安全原因排除了支付详情。数据仅由查看自己历史的已认证用户访问，由 `src/middleware/auth.js:34` 的中间件强制执行。这告诉你对于新功能什么是可能的，以及适用什么约束。

### 模式 3："当 [条件] 发生时会发生什么？"

边缘情况和错误场景对产品质量很重要，但如果不看实现就很难发现。支付在结账中途失败会发生什么？用户会话在表单提交期间过期会发生什么？在加入购物车到完成购买之间库存耗尽会发生什么？

> **提示词：边缘情况调查**
>
> 当结账期间支付失败时会发生什么？请带我走一遍：
> - 用户看到什么（错误消息、页面状态）
> - 他们的购物车和订单会发生什么
> - 我们会重试还是他们必须重新开始
> - 我们如何处理部分完成（订单已创建但支付失败）

Claude Code 读取错误处理代码、支付集成逻辑和状态管理来解释行为。你发现支付失败显示通用错误、保留购物车、允许重试而不丢失进度。或者你发现了边缘情况未得到良好处理的缺口，这些成为待办事项。

### 模式 4："为什么会出现 [行为]？"

用户报告令人困惑或违反直觉的行为。在分类之前你需要知道这是设计如此还是 bug。为什么搜索过滤器在点击"显示更多"时会重置？为什么时间戳显示 UTC 而非用户时区？为什么表单接受无效电话号码？

> **提示词：行为解释**
>
> 用户报告搜索过滤器在点击"显示更多"加载更多结果时会重置。为什么会这样？是设计如此还是 bug？
>
> 解释代码中什么导致此行为，以及是否有产品上的原因。

你了解到"显示更多"触发了整页刷新（遗留实现），这会重置包括过滤器在内的所有组件状态。这不是有意的，而是旧代码的技术限制。现在你可以将其作为 bug 进行分类，并提供根本原因的上下文给工程团队。

### 让这些模式发挥作用

- **提示词要具体。**"认证是如何工作的？"太宽泛。Claude Code 可能读取 20 个文件给你一个系统概览，而你只需要知道一个细节。"当用户访问受保护页面时，应用如何确定用户是否已登录？"聚焦且得到直接答案。
- **请求产品语言。** 始终包含"用产品术语解释"或"聚焦于面向用户的行为"，以防止你无法解析的代码密集型解释。除非你指定，Claude Code 默认提供技术细节。
- **提出后续问题。** 第一个答案给你方向。后续问题深入挖掘："你提到了第 92 行的业务规则。那个规则的设计推理是什么？"或"你说支付失败会保留购物车。那个逻辑在哪里实现？"
- **引用文件位置。** Claude Code 提供文件路径和行号。在需要工程帮助时使用这些："我调查了折扣码问题，在 `discount-validator.js:92` 找到了逻辑。你能解释一下阻止促销叠加的设计推理吗？"这比"嘿，折扣码怎么工作的？"有效得多。

这四种模式处理 90% 的功能调查。你不在阅读代码，你在提出有依据的问题，并将实现细节翻译成产品上下文。调查可能耗时 10-20 分钟，花费 $0.15-0.50，但它回答了那些本来需要打断工程团队或对实现现实保持无知的问题。

## 3.4 不读代码就能读懂代码

你可以在不解析语法的情况下理解实现。Claude Code 将代码翻译成概念、图表和简明英文。本节介绍为你无法直接阅读的代码库构建心智模型的技巧。

**请求简明英文摘要。** 代码精确但不透明。自然语言可解释但可能模糊。让 Claude Code 弥合差距：

> **提示词：简明英文摘要**
>
> 用简明英文总结 `processCheckout` 函数的功能。聚焦于：
> - 它按顺序执行的步骤
> - 它需要什么输入
> - 它返回或产生什么
> - 什么可能出错以及它如何处理错误

输出："该函数接收一个购物车对象和支付信息。首先，验证购物车有商品且总额正确。其次，通过 Stripe 集成处理支付。如果支付成功，在数据库中创建订单记录，发送确认邮件，并清空购物车。如果支付失败，记录错误并返回失败，不创建订单。如果数据库写入在成功支付后失败，将订单加入队列等待手动对账。"

你现在理解了结账流程，没有读一行代码。你可以向利益相关者解释，识别改进机会，并就边缘情况提出有意义的问题。

**请求流程图和时序图。** 可视化表示通常比文本更清晰地阐明复杂逻辑。Claude Code 无法生成实际图像，但可以生成文本图表展示流程：

> **提示词：流程图请求**
>
> 创建一个文本流程图，展示用户提交结账表单时发生的情况。显示决策点、成功/失败路径以及外部服务调用。

输出可能是：
```
用户提交结账表单
        ↓
验证购物车（商品存在、价格当前）
        ├─ 无效 → 向用户返回错误
        └─ 有效 → 继续
                    ↓
          通过 Stripe API 处理支付
                    ├─ 支付失败 → 记录错误，显示用户消息，保留购物车
                    └─ 支付成功 → 继续
                                ↓
                        在数据库中创建订单
                                ├─ 数据库错误 → 加入手动对账队列，通知团队
                                └─ 成功 → 继续
                                        ↓
                                发送确认邮件
                                        ├─ 邮件失败 → 记录警告（订单仍然已创建）
                                        └─ 成功 → 继续
                                                ↓
                                清空购物车，重定向到确认页面
```

这向你展示了每个决策点、失败模式和依赖。你可以走查客户场景。"如果支付成功但邮件发送失败怎么办？"图表显示订单已创建，邮件失败被记录但不阻止完成。这是重要的产品知识。

**识别功能的"重要"文件。** 大型代码库有数百或数千个文件，大多数与任何给定功能无关。让 Claude Code 过滤：

> **提示词：关键文件识别**
>
> 我需要了解结账功能。我应该了解的最重要的五个文件是什么？对每个文件，解释它负责什么以及为什么对结账很重要。

Claude Code 分析代码库并排序优先级："1. `CheckoutForm.tsx` ——用户交互的 React 组件，处理表单提交并显示错误。2. `checkout.service.js` ——包含处理结账的业务逻辑，验证购物车并编排支付。3. `stripe-integration.js` ——处理所有用于支付处理的 Stripe API 调用。4. `order.model.js` ——订单的数据库模型，定义我们存储什么数据。5. `email-notifications.js` ——发送确认邮件并处理模板渲染。"

现在当你调查结账问题或规划改进时，你知道该看哪里。你不是盲目导航，而是带着上下文开始。

**在没有语法知识的情况下构建心智模型。** 你的目标不是阅读代码，而是理解功能如何表现、数据如何流动以及决策在哪里做出。将问题聚焦于概念：

- "用户注册过程中的主要步骤是什么？"
- "应用如何决定向用户显示哪个定价层级？"
- "搜索功能依赖什么外部服务？"
- "我们在哪里存储用户偏好，它们如何被加载？"

这些问题揭示了系统行为和架构，无需你解析函数语法、理解循环或追踪变量赋值。Claude Code 进行解析并给你概念模型。

**AI 生成解释的局限性。** Claude Code 根据所读内容解释代码，但无法运行代码或保证其解释正确。如果逻辑复杂、存在微妙 bug 或依赖于 Claude Code 无法看到的运行时条件，解释可能不完整或错误。

将解释视为高置信度假设，而非绝对真理。当精确性重要时（调试关键问题、评估安全影响、规划重大变更），与工程团队验证。当你需要一般性理解时（回答利益相关者问题、分类 bug、评估可行性），Claude Code 的解释足够可靠，可以作为行动依据。

你通过提出正确的问题并接受摘要而非语法，正在"不读代码就读懂代码"。这对 80% 的 PM 需求是足够的。剩下的 20%（深度调试、安全审计、性能优化）仍需要工程专业知识。知道边界，你就不会越界。

## 3.5 让 Claude Code 记住你的上下文

每个 Claude Code 会话开始时，如果项目中存在 `CLAUDE.md`，它会自动读取。这个文件包含跨所有会话持久化的上下文：你的项目机构知识，不属于代码注释或外部文档的内容。

对于 PM 来说，维护良好的 `CLAUDE.md` 将 Claude Code 从智能助手转变为了解你产品的团队成员。没有 `CLAUDE.md` 的第一个会话需要解释你的领域。有了 `CLAUDE.md`，每个会话都在知情基础上开始。设置时间立即回本。

**目的：跨会话存活的持久化上下文。** Claude.ai 在你关闭窗口时忘记一切。Claude Code 在你退出会话时忘记对话历史。但 `CLAUDE.md` 持久存在。你希望 Claude Code 在每个会话中都知道的任何内容放在这里。

这不是所有文档的倾倒场，而是能够提高调查质量、减少重复解释的聚焦上下文。把它想象成 AI 团队成员的入职文档。

**为 PM 工作流应包含的内容：**

**产品领域术语表。** 你的产品有模糊或领域特定的术语。一次定义：
```
## 产品术语
- **Credit（积分）**：用户通过推荐获得的内部货币。与支付信用无关。
- **Workspace（工作区）**：团队的共享环境。用户可属于多个工作区。
- **Pipeline（管道）**：销售管道，非数据管道。指 CRM 中的交易阶段。
- **Activation（激活）**：用户完成个人资料设置并创建第一个项目。与账户激活（邮箱验证）不同。
```
这防止 Claude Code 误解具有多重含义的术语。当你询问"activation rates"时，它知道你指的是个人资料完成，而非邮箱验证。

**映射到代码区域的关键用户旅程。** 将产品流程连接到实现：
```
## 用户旅程
### 新用户引导
1. 用户注册 → `src/auth/signup.js`
2. 邮箱验证 → `src/services/email-verification.js`
3. 个人资料设置 → `src/components/ProfileSetup.tsx`
4. 创建第一个项目 → `src/services/project-creator.js`

### 结账流程
- 入口：`src/components/Checkout/CheckoutForm.tsx`
- 支付处理：`src/services/payment-processor.js`
- 订单创建：`src/models/order.js`
- 确认：`src/services/order-notifications.js`
```
当你让 Claude Code 调查引导流程的某部分时，它无需探索就知道该看哪里。这节省了 token 和时间。

**团队约定和沟通规范。** 你的团队如何工作？Claude Code 应该了解你们的哪些实践？
```
## 团队约定
- 所有面向客户的字符串使用 `src/locales/` 中的 i18n 系统。不要建议硬编码字符串。
- 定价逻辑仅存在于 `src/services/pricing/` 中。前端从不计算价格。
- 我们在提交信息中使用 Jira 工单 ID："Fix checkout bug (PM-1234)"
- 数据库迁移在运行前需要数据团队批准
- 功能标志在 `src/config/features.js` 中管理
```
这防止 Claude Code 建议违反团队实践的变更。如果你请求帮助生成发布说明，它知道在提交中查找 Jira ID。

**外部文档链接。** 指向 Claude Code 无法访问但你经常参考的资源：
```
## 外部资源
- API 文档：https://docs.internal.com/api
- 设计系统：https://www.figma.com/design-system
- 产品策略文档：[Google Docs 链接]
- 分析仪表盘：https://analytics.internal.com
讨论功能时，考虑与设计系统的一致性以及与产品策略的对齐。
```

**放置位置：CLAUDE.md 放在哪里。** 主要和推荐的位置是你的项目根目录：`/path/to/your/repo/CLAUDE.md`。这使文件对浏览仓库的任何人可见，并将其确立为团队文档。

Claude Code 以层级方式读取 CLAUDE.md 文件。首先从你的主目录（`~/.claude/CLAUDE.md` 用于跨所有项目的个人偏好），然后从你的项目根目录。你还可以在子目录中放置 CLAUDE.md 文件，用于特定于代码库该部分的上下文，尽管这不常见。

大多数 PM 使用场景倾向于在项目根目录中使用单个 CLAUDE.md。你添加的内容（领域知识、用户旅程映射、约定）不仅对 Claude Code 有益，也让工程师和设计师受益。让它可见，并将其视为团队共同维护的活文档。

**随着你的理解增长更新 CLAUDE.md。** 将此文件视为活文档。当你调查功能并学到有价值的内容时，添加进去。当团队约定改变时，更新它。当你注意到 Claude Code 反复犯同样的错误（误解术语、遗漏模式）时，添加澄清。

当这个文件阻止了一次重复解释，或帮助新 PM 通过阅读你的 CLAUDE.md 来理解产品上下文时，它就收回了成本。版本控制追踪变更，所以你可以看到理解如何随时间演变。

**面向 PM 的 CLAUDE.md 示例结构：**

```markdown
# Claude Code 产品上下文
## 关于本产品
[2-3 句话：这个产品做什么？谁在使用它？]
## 产品术语
[关键术语及其在领域中的具体含义]
## 用户旅程
[映射到代码位置的关键用户流程]
## 业务规则
[从代码中不明显的重要产品逻辑]
## 团队约定
[我们如何工作，避免建议的内容，应遵循的实践]
## 外部资源
[指向文档、设计、策略、分析的链接]
## 常见 PM 问题
[经常调查的主题及快速答案或指针]
```

从小处开始。添加术语表和一两个关键用户旅程。随着你发现哪些上下文能提高调查质量而逐步扩展。一个 50 行、内容聚焦且高价值的 CLAUDE.md 胜过 500 行把你所知一切都倒进去的文件。

## 3.6 知道何时升级给工程团队

Claude Code 读取代码、解释并以简明英文说明。这对大多数 PM 需求工作得非常好，但它有你需认识的边界。对 AI 生成的解释过度自信会在你基于不完整或不正确的信息行动时产生问题。

**运行时行为 vs. 静态分析。** Claude Code 读取源代码：应用的静态文本。它无法运行代码、观察实际行为或查看运行时状态。当行为依赖于配置、环境变量、数据库状态或外部服务响应时，这一点很重要。

具体例子：你问应用如何确定用户可以访问哪些功能。Claude Code 读取权限检查代码并解释逻辑："应用检查 `user.role` 是否为 'admin'，如果是则授予管理功能访问权限。"但它无法告诉你实际在生产环境中分配了什么角色，是否有数据质量问题导致错误的角色值，或环境变量是否在某些部署中覆盖了此逻辑。

对于理解某事应该如何工作，Claude Code 是可靠的。对于理解它现在在生产环境中实际如何工作，你需要运行时数据：日志、数据库查询、监控仪表盘或工程帮助。

**配置和环境依赖。** 应用基于配置文件、环境变量和部署上下文表现不同。开发、预发布和生产环境可能运行相同代码但行为因配置而异。

Claude Code 可以读取配置文件并识别依赖，但除非明确告知，它无法告诉你哪个配置在哪个环境中是活动的。如果你问"我们如何处理支付失败？"而答案取决于 Stripe 处于测试模式还是生产模式，Claude Code 解释两条代码路径，但可能不会澄清哪个适用于哪里。

在调查对环境敏感的功能时，明确说明配置。"我们在生产环境中如何处理支付失败？"提示 Claude Code 查找生产配置。

**何时升级给工程团队。** 你可以独立调查，但某些情况需要工程专业知识：

- **安全敏感问题。**"这个认证实现安全吗？"或"用户是否能访问其他用户的数据？"需要安全专业知识，而不仅仅是代码阅读。Claude Code 可以识别明显问题（明文存储密码、明显的 SQL 注入漏洞），但微妙的安全问题需要人工审查。
- **性能问题。**"为什么这个功能很慢？"涉及性能分析、数据库查询分析和运行时行为观察。Claude Code 可以通过阅读代码识别潜在的昂贵操作（嵌套循环、N+1 查询），但实际性能取决于数据量、数据库索引和它看不到的基础设施。
- **复杂调试。** 如果 bug 是间歇性的、依赖数据的或涉及竞态条件，Claude Code 的静态分析无法找到根本原因。用它进行初始调查（缩小相关代码区域的范围），然后将带上下文的报告交给工程团队。
- **架构决策。**"我们应该将其重构为微服务吗？"或"这对我们规模来说是正确的数据库吗？"需要工程判断，涉及权衡、团队能力和长期维护。Claude Code 可以解释当前架构并识别痛点，但无法做出战略性技术决策。
- **生产系统访问。** 任何需要数据库查询、日志分析或监控仪表盘审查的事情都超出 Claude Code 的范围。它处理代码和文件，而非实时系统。

**避免对解释的过度自信。** 将 Claude Code 的解释视为高置信度解释，而非绝对真理。当准确性重要时（面向客户的沟通、路线图决策、安全评估），在行动前与工程团队验证发现结果。

使用这个心智模型：Claude Code 像一个能阅读代码并解释所见内容的初级工程师，但没有生产经验，也没有与原开发者交流过。你相信他们的阅读，但验证任何关键内容。

**PM 使用的实际边界：**

**图 3.3：升级边界** ——用于知道何时独立调查、何时让工程团队参与。

| 你可以用 Claude Code 处理 | 升级给工程团队 |
|---------------------------|----------------|
| 功能 X 如何工作？ | 功能 X 安全吗？ |
| 哪些文件实现 Y？ | 为什么 Y 在生产中很慢？ |
| Z 访问什么数据？ | 我们应该重构 Z 吗？ |
| 配置 A 在哪里定义？ | 生产环境中配置 A 是什么？ |
| 用户做 B 时会发生什么？ | 为什么 B 对某些用户失败？ |
| 能否组合功能 C 和 D？ | 我们应该如何构建 C+D 的集成？ |
| E 的边缘情况有哪些？ | 我们如何在预发布环境测试 E？ |

模式是：关于**什么存在以及它是如何结构的**问题是 Claude Code 的领域。关于**质量、性能、正确性和生产行为**的问题需要工程团队。

这并不贬低 Claude Code 的价值，而是将你的使用聚焦于高回报的调查，同时对 AI 的局限性保持适当的谦逊。你仍然会通过自己处理第一轮调查节省数小时的工程时间。你只是避免了将 AI 解释当作神谕真理的陷阱。

---

你现在可以在不依赖工程团队回答每个问题的情况下调查代码库。你理解了如何通过架构概览定位自己、应用功能调查模式、在不读代码的情况下构建心智模型、在 CLAUDE.md 中创建持久上下文、以及识别何时需要升级。第 4 章将向你展示如何将这些能力具体应用于 bug 调查——这是代码库访问能带来即时、可衡量价值的 PM 工作流。
# 第 4 章 调查 Bug 和用户问题

## 4.1 无需工程帮助即可分类的内容

一个支持工单进来了："我的支付没通过，但我被收费了。"工程团队正在发布一个版本。客户正在升级投诉到你的 VP。你需要分类：真实的 bug、用户操作失误还是边缘情况？你需要评估范围：一个客户还是系统性的？你需要上下文：工程团队在调查前应该知道什么？

传统 PM 的回应：将工单转发给工程团队，附上一句"你能看一下吗？"然后等待。工程师切换上下文、调查、反馈。如果是用户操作失误，你白白打断了 sprint。如果是真实 bug，除了原始客户投诉你没有任何上下文可以提供。无论哪种方式，反馈循环需要数小时或数天。

替代方案：打开 Claude Code，自己调查支付流程，识别可能导致此行为的代码路径，在分配给工程团队之前整理出调查结果。总耗时：20 分钟。工程团队收到一份聚焦的调查，附有文件引用和假设。他们确认并修复，无需发现阶段。

Bug 调查是 Claude Code 为 PM 带来即时、可衡量价值的场景。你不是在修 bug，那是工程团队的工作。你是在收集上下文、形成假设并减少来回沟通。好的分类每周为工程团队节省数小时。糟糕的分类用不完整的交接浪费所有人的时间，还需要跟进问题。

**你在 bug 调查中的角色：**

- **这是真实 bug 还是用户操作失误？** 很多"bug"是用户不理解预期行为、配置错误或会话过期。调查通常会揭示系统工作正常而用户理解有误。发现这一点为工程团队节省了一个浪费的周期。
- **范围和严重程度如何？** 影响一个客户还是多个？涉及数据丢失还是外观问题？间歇性还是可复现？这些上下文决定优先级。Claude Code 帮助你理解什么条件触发这个问题，以及这些条件可能有多广泛。
- **工程团队需要什么上下文？** 文件位置、相关代码路径、类似的过往问题、你已识别的复现步骤。你提供的上下文越多，工程团队解决问题的速度就越快。没有调查的原始客户投诉会产生更多工作，而非更少。

你不被期望修复 bug。你被期望进行智能分类、提供有用的上下文、并避免将工程时间浪费在你自己本可以解决或澄清的问题上。Claude Code 使这在不读代码的情况下成为可能。

## 4.2 将用户投诉翻译为可测试的查询

用户语言与代码语言不匹配。客户描述症状："页面坏了。""我的折扣没有应用。""我无法登录。"工程师需要具体信息：错误消息、复现步骤、环境细节、数据状态。你的工作是翻译。

**将用户语言翻译为技术查询。** 从模糊描述中提取具体信息。报告说"结账页面坏了。"具体坏在哪里？他们看到错误消息了吗？他们当时在尝试做什么？实际发生了什么，与预期有何不同？

如果你有原始支持工单的访问权限，提取这些细节。如果没有，做出合理的假设并在调查中注明。然后让 Claude Code 帮助识别什么可能导致描述的症状。

以 plan 模式启动 Claude Code：

```bash
claude --permission-mode plan
```

> **提示词：症状翻译**
>
> 一个用户报告："[粘贴确切的投诉内容]"
>
> 什么代码路径可能导致此行为？请识别：
> - 哪些文件处理此功能
> - 每个步骤可能出现什么问题
> - 用户可能遇到什么错误状态
> - 什么数据或条件可能触发此问题

例如，对于"我的折扣码没有应用，尽管它是有效的"：

> 一个用户报告："我的折扣码没有应用，尽管它是有效的。我输入正确，它只是显示'invalid code'。"
>
> 在折扣或结账流程中，什么代码路径可能导致此行为？识别潜在的失败点：代码验证逻辑、过期检查、使用限制、购物车要求、大小写敏感。

Claude Code 读取相关文件并返回类似如下内容："折扣验证发生在 `src/services/discount-validator.js`。代码检查：(1) 代码在数据库中存在，(2) 基于 `valid_until` 日期未过期，(3) 使用计数低于 `max_uses`，(4) 购物车满足 `minimum_cart_value`。任何一个检查失败都返回 'invalid code'，没有具体信息。大小写敏感被强制执行，所以 'SUMMER20' 和 'summer20' 是不同的代码。"

你现在有了一个可测试的假设。通用的"invalid code"消息掩盖了实际的失败原因。客户的代码可能已过期、超过使用限制或大小写输入错误。所有这些对用户来说看起来是有效的，但被系统拒绝了。

**识别相关的日志模式和错误消息。** 如果你的应用记录了错误，让 Claude Code 查看存在什么日志：

> 折扣码验证存在什么日志或错误追踪？我在哪里可以找到折扣应用失败的日志？

Claude Code 识别日志语句、错误处理器以及错误详情被捕获的位置。你了解到失败的验证记录了 `logs/discount-errors.log` 及原因代码，或者失败根本没有被记录。缺失日志本身就是改进调试的有用信息。

**区分前端 vs. 后端 vs. 数据问题。** 不是所有 bug 都是代码 bug。有些是配置问题、数据质量问题或环境不匹配。

> **提示词：问题层级识别**
>
> 用户经历了 [症状]。帮我确定这可能是：
> - 前端问题（显示、客户端逻辑、浏览器兼容性）
> - 后端问题（服务端逻辑、API 响应、业务规则）
> - 数据问题（数据库状态、用户记录问题、数据缺失）
> - 配置问题（环境设置、功能标志、第三方服务）
>
> 什么证据可以区分这些？

对于折扣码的例子，Claude Code 可能解释："前端问题，如果代码没有被正确发送的话（检查网络请求）。后端问题，如果验证逻辑有 bug 的话（我们审查的代码）。数据问题，如果数据库中的折扣码记录格式错误或缺失的话。配置问题，如果此环境中折扣服务被禁用或 Stripe 没有正确处理的话。"

这缩小了你的调查范围。如果客户说"我输入了代码立即收到错误"，这很可能是前端或后端验证。如果他们说"似乎工作了但折扣没有显示在收据上"，这很可能是稍后的处理步骤，也许是后端计算或数据持久化。

**复现调查的成本和时间。** 一次典型的复现调查会话（翻译用户投诉、识别相关代码、缩小到可能原因）耗时 10-20 分钟，消耗 15,000-40,000 token。API 用户成本：$0.08-0.20。订阅用户成本：已包含。对比替代方案：一个模糊的工单被工程团队退回要求更多信息，增加一天的解决时间。

## 4.3 30 分钟内从用户报告到工程交接

结构化调查胜过临时探索。遵循此工作流，从用户报告进展到可操作的工程交接。

**图 4.1：Bug 调查工作流** ——用于结构化你的 bug 分类过程。五个步骤：1) 收集用户报告详情（5 分钟），2) 用 Claude Code 识别相关代码区域，3) 追踪代码路径，4) 形成可测试的假设，5) 记录发现结果用于工程交接。总耗时：20-40 分钟。

**步骤 1：收集用户报告详情。** 在接触 Claude Code 之前，从报告中提取一切你能提取的：
- 用户当时正在尝试做什么
- 他们期望发生什么
- 实际发生了什么
- 他们看到的任何错误消息（如有可能，精确文本）
- 何时发生
- 他们使用什么浏览器/设备/环境
- 他们能否复现还是一次性的

这需要五分钟，可以防止浪费的调查周期。如果你在不知道用户使用的是过时浏览器的情况下调查"登录坏了"，你将追踪正常工作的代码。

**步骤 2：让 Claude Code 识别相关代码区域。** 收集好上下文后，开始你的调查会话：

```bash
claude --permission-mode plan
```

> **提示词：代码区域识别**
>
> 我正在调查一个用户报告的问题：[一句话摘要]
>
> 上下文：
> - 用户操作：[他们在做什么]
> - 预期：[应该发生什么]
> - 实际：[实际发生了什么]
> - 错误消息（如有）：[精确文本]
>
> 识别可能涉及的代码区域。请展示：
> - 入口点（此用户操作在代码中从哪里开始）
> - 执行路径中的关键文件
> - 错误在哪里被捕获和处理
> - 这可能在哪里失败

Claude Code 绘制出相关的代码领域。对于支付失败调查，你可能得到："用户点击支付 → `CheckoutButton.tsx` 派发操作 → `checkout.service.js` 验证购物车 → `payment-processor.js` 调用 Stripe API → 响应在 `payment-response-handler.js` 中处理 → 订单在 `order.service.js` 中创建。每一步都存在错误处理；Stripe 集成在 `stripe-errors.js` 中有特定的错误映射。"

**步骤 3：请求解释可疑的代码路径。** 一旦知道该看哪里，请求细节：

> 逐步解释 `payment-processor.js` 做什么。聚焦于：
> - 什么可能导致它失败
> - 当 Stripe 返回错误时会发生什么
> - 用户看到具体错误消息还是通用消息
> - 支付失败时什么被记录

Claude Code 读取文件并翻译："处理器验证支付方式存在，检查购物车总额与收费金额匹配，调用 Stripe 的 charge API，处理响应。Stripe 错误被捕获并映射。'card_declined' 变成 'Your card was declined'，'insufficient_funds' 出于隐私变成相同消息。所有失败将 Stripe 错误代码记录到 `payments.log`，但向用户显示通用消息。如果 Stripe 返回成功但之后订单创建失败，有一个对账队列捕获孤立的支付。"

现在你有了进展。用户说他们被收费但支付"没有通过"。这可能意味着：Stripe 收费成功，但订单创建失败，触发对账队列。客户被收费但看到错误，因为失败发生在支付之后。

**步骤 4：形成假设。** 基于你的调查，形成可测试的假设：

"用户很可能被 Stripe 成功收费，但后续订单创建失败。这可能是数据库错误、超时或验证问题。系统在支付成功后显示通用错误消息。对账队列应包含此案例。如果是这样，支付存在于 Stripe 中，用户需要手动创建订单或退款。"

这是可操作的。工程团队可以检查对账队列，在 Stripe 中验证，并解决具体案例，同时调查订单创建失败的原因。

**步骤 5：为工程团队记录发现结果。** 整理你的调查：

> **提示词：调查摘要**
>
> 将我对这个 bug 的调查总结成工程交接文档。包括：
> - 用户报告了什么
> - 我调查了什么
> - 我审查的文件和代码区域
> - 我对根本原因的假设
> - 给工程团队的具体问题
> - 建议的下一步

Claude Code 生成一份你可以粘贴到 Jira、Slack 或邮件中的摘要。添加它找到的文件引用、你的假设以及工程团队应验证的内容。这种交接尊重工程时间。他们不是从零开始。

**完整工作流耗时**：中等复杂度的 bug 为 20-40 分钟。Token 消耗：30,000-80,000 token。对于 API 用户，这花费 $0.15-0.40。一次为工程团队节省一小时发现时间的彻底调查，其价值远超于此。

## 4.4 在工程团队之前识别 Bug 类型

某些 bug 类别在各类产品中反复出现。识别这些模式帮助你更快调查，更精确地与工程团队沟通。你不是在调试代码，而是在将症状与已知问题类型进行模式匹配。

**图 4.2：Bug 模式参考** ——用于快速分类用户报告的问题。六种常见模式：验证缺口（无效数据被保存）、状态管理（旧数据出现）、竞态条件（重复扣费）、配置不匹配（部分用户正常工作）、集成失败（支付意外失败）、访问控制（看到其他用户数据）。将症状匹配到模式以加速调查。

**数据验证缺口。** 不应通过的用户输入通过了。表单接受无效的电子邮件地址。价格字段允许负数。必填字段可以被绕过。这些 bug 源于不完整或前后端不一致的验证逻辑。

> **提示词：验证调查**
>
> [字段/输入] 在此功能中是如何被验证的？检查：
> - 前端验证（表单在提交前检查什么）
> - 后端验证（服务器在接收数据时检查什么）
> - 数据库约束（schema 强制执行什么）
> - 是否有无效数据可以通过的缺口？

Claude Code 比较每一层的验证。你可能发现前端检查邮件格式但后端不检查，允许 API 调用绕过前端验证。这是你的假设：症状（数据库中的无效数据）匹配模式（验证缺口）。

**状态管理问题。** UI 显示过时数据。更改在导航后不持久。用户看到别人的数据。这些 bug 涉及应用如何追踪和同步状态。

> **提示词：状态调查**
>
> [功能] 的状态是如何管理的？追踪：
> - 状态存储在哪里（本地、会话、服务器）
> - 用户操作后状态如何更新
> - 状态何时刷新或同步
> - 什么可能导致状态过时或不正确

状态 bug 很微妙，因为代码"能运行"但就是协调不正确。Claude Code 可以识别用户资料被本地缓存但在服务端更新，且缓存未在更新时失效。症状匹配模式。

**竞态条件（概念识别）。** 两个不应同时发生的操作碰撞了。用户点击按钮两次被重复扣费。并发编辑互相覆盖。这些是需要特定条件才能触发的时序 bug。

你无法用静态分析调试竞态条件，但可以识别易感性：

> [功能] 可能存在竞态条件吗？寻找：
> - 触发异步操作的用户操作
> - 没有点击保护的按钮或表单
> - 对共享资源的并发访问
> - 应该是原子操作但实际上不是的操作

Claude Code 可能发现："结账按钮触发异步支付处理但在点击时不禁用。多次点击可以派发多次支付调用。Stripe 调用中没有幂等键防止重复扣费。"

你无法在没有运行时测试的情况下确认这是根本原因，但你已经识别了模式。你的 bug 报告说："疑似竞态条件。结账按钮缺乏点击保护，支付调用可能不幂等。用户可能双击了。"

**配置不匹配。** 功能在某些环境中工作但在其他环境中不工作。预发布环境行为与生产环境不同。某些用户体验到其他人没有的问题。这些 bug 源于环境特定的设置。

> **提示词：配置调查**
>
> 什么配置控制 [功能]？展示：
> - 影响行为的环境变量
> - 功能标志或开关
> - 环境间不同的设置
> - 用户特定或账户特定的设置

你了解到该功能使用一个第三方 API，每个环境有不同的凭证，而生产凭证已过期。或者一个功能标志仅为 beta 用户启用该功能，而受影响的用户不在 beta 组中。

**第三方集成失败。** 外部服务返回错误、超时或行为异常。支付处理器拒绝有效卡片。邮件服务静默失败。分析事件不出现。

> **提示词：集成调查**
>
> 此功能如何与 [第三方服务] 集成？展示：
> - API 调用在哪里发出
> - 来自服务的错误如何处理
> - 失败是否以及如何被记录
> - 集成失败时用户看到什么

Claude Code 追踪集成。你发现 Stripe 错误被记录但用户看到通用"出了点问题"消息。或者邮件发送是 fire-and-forget 模式没有错误处理，所以失败无声消失。

**权限和访问控制 bug。** 用户可以访问不应访问的内容。用户无法访问应该能访问的内容。管理员专属功能对普通用户出现。

> **提示词：访问控制调查**
>
> [功能] 的权限是如何检查的？展示：
> - 访问控制逻辑在哪里
> - 什么条件授予访问权限
> - 系统如何确定用户角色/权限
> - 访问被拒绝时发生什么

你发现权限检查仅在前端发生，所以用户可以通过直接 API 调用绕过。或者权限检查有一个应该是 AND 的 OR。症状匹配模式：访问控制逻辑有缺陷。

**模式参考表：**

| 用户报告... | 可能的模式 | 调查焦点 |
|-------------|-----------|---------|
| "无效数据被保存了" | 验证缺口 | 比较前端/后端/数据库验证 |
| "旧数据出现了"/"更改没保存" | 状态管理 | 追踪状态存储、更新、同步 |
| "被重复扣费了" | 竞态条件 | 寻找无保护的异步操作 |
| "部分用户能用但我不能" | 配置不匹配 | 检查用户特定或环境设置 |
| "支付失败了但应该能成功" | 第三方集成 | 追踪外部 API 调用和错误处理 |
| "我能看到其他用户的数据" | 访问控制 | 审查权限逻辑和执行 |

识别模式将调查从"可能是什么问题"加速到"是这个具体问题吗？"你不是在调试，而是在将症状分类以与工程团队精确沟通。

## 4.5 工程团队真正想读的 Bug 报告

你的调查只有在有效沟通时才有价值。工程团队需要具体信息才能快速行动。一份附有调查上下文的良好结构 bug 报告将解决时间从天缩短到小时。

**工程团队需要什么：**

- **复现步骤。** 他们如何触发问题？从起始状态到错误的编号步骤。"1. 创建新用户账户。2. 将商品加入购物车。3. 应用折扣码 'SUMMER20'。4. 点击结账。5. 观察到错误消息。"如果你无法复现，说明情况并描述用户报告的内容。
- **环境。** 在哪里发生？生产/预发布/开发。浏览器和版本。设备类型。用户账户类型（新 vs. 现有，免费 vs. 付费）。任何可能影响行为的因素。
- **预期 vs. 实际。** 应该发生什么？实际发生了什么？要具体。"预期：折扣应用，购物车总额更新。实际：出现错误消息 'Invalid code'，尽管代码在管理面板中有效。"
- **错误消息。** 精确文本，而非转述。错误代码（如果可见）。用户提供的截图。如果你有访问权限，日志条目。

**使用 Claude Code 输出丰富报告。** 你的调查产生了文件引用、代码解释和假设。将它们纳入：

> **提示词：Bug 报告生成**
>
> 基于我的调查，生成一份 bug 报告，包括：
> - 问题摘要
> - 复现步骤（基于我的理解）
> - 环境细节
> - 技术上下文（涉及的文件、代码路径）
> - 我对根本原因的假设
> - 建议工程团队的调查方向
> - 我无法回答的开放问题

Claude Code 将会话综合为结构化输出。你仍需要审查和编辑（添加它未看到的用户报告细节、删除你不确定的推测），但它创建了一个良好的起始草稿。

**在不假装有专业知识的情况下包含相关代码片段。** 你发现 `discount-validator.js:45-78` 中的折扣验证可能导致此问题。在不假装理解每一行的情况下包含引用：

"我调查了折扣验证流程，在 `discount-validator.js:45-78` 找到了逻辑。根据我的阅读，代码检查 [列出条件]。用户的场景可能在 [特定条件] 失败，因为 [假设]。我不确定这就是根本原因，但这是我首先会看的地方。"

这提供了有用的上下文而不越界。你在说"我看了这里，这似乎相关"，而不是"我确切知道哪里有问题。"工程师欣赏上下文，不会反感适当的谦逊。

**模板：经 PM 调查的 Bug 报告：**

```
## 摘要
[一句话描述问题]

## 用户报告
- **用户操作**：[他们正在尝试做什么]
- **预期**：[应该发生什么]
- **实际**：[实际发生了什么]
- **错误消息**：[精确文本，如有]

## 环境
- **位置**：生产 / 预发布 / 等
- **用户类型**：[账户特征]
- **浏览器/设备**：[如已知]
- **时间**：[日期/时间，如相关]

## 复现步骤
1. [步骤 1]
2. [步骤 2]
3. [观察：具体行为]
（注明复现是由 PM 确认还是根据用户报告推断）

## PM 调查
### 审查的文件
- `path/to/file.js` — [此文件处理什么]
- `path/to/other.js` — [此文件处理什么]

### 发现
[你从阅读代码中学到了什么。流程是什么样的。在哪里可能失败。]

### 假设
[你对根本原因的最佳猜测，以假设而非结论的形式陈述]

### 开放问题
- [你无法从代码中确定的事项]
- [需要运行时验证的事项]

## 建议的下一步
- [ ] 在日志中验证 [具体事项]
- [ ] 检查 [具体状态或数据]
- [ ] 测试 [具体场景]

## 优先级评估
[你的建议：P1/P2/P3 及原因。范围评估——一个用户还是多个？]
```

**不要包含什么。** 不要包含你不理解的大段代码。不要将假设陈述为结论。不要归咎于特定提交或工程师。不要评论修复有多"明显"。坚持观察、假设和问题。

这样的 bug 报告将模糊的"什么东西坏了"变成了可操作的工程工作。你已经完成了前 30% 的调查。工程团队完成剩余的 70%，但这部分工作不再是从零开始，而是聚焦和高效的。

## 4.6 知道何时停止

调查有递减回报。在某个点上，进一步的 PM 调查浪费你的时间并延迟工程团队的介入。识别边界。

**你已经收集了足够上下文的标志：**

- **你能解释流程。** 你理解功能做什么、数据如何在其中流动以及它在代码中的哪里发生。即使你无法流畅阅读代码，你也可以描述从用户操作到结果的路径。
- **你有一个可测试的假设。** 你已经形成了一个关于出错原因的合理猜测。"折扣验证失败是因为代码大小写敏感，而用户输入了小写。""支付成功但订单创建因数据库超时失败。"假设不需要是对的——它们给工程团队一个起点。
- **你已经识别了相关文件。** 你可以指向问题可能存在的具体文件路径。这为工程团队节省了探索阶段。
- **你知道你不知道什么。** 你的调查遇到了限制：你看不到的运行时行为、你无法验证的数据状态、你无法访问的配置。记录这些缺口帮助工程团队聚焦调查。

如果你已经达到这四个标准，就够了。写 bug 报告并交接。

**你力不能及的标志：**

- **代码太复杂无法总结。** 如果 Claude Code 的解释即使是用简明英文也不合理，逻辑可能超出了调查所能澄清的范围。交接给工程团队。
- **你在打转。** 你已经将同一个问题的不同变体问了三次，没有获得新信息。问题需要运行时调试，而非代码阅读。
- **涉及安全敏感问题。** 认证、授权、数据访问、加密：任何搞错了后果严重的事情。你的假设可能危险地错误。工程安全审查是必需的。
- **需要生产环境访问。** 检查数据库状态、阅读生产日志、用真实用户数据测试。Claude Code 读取代码，而非实时系统。你已触及边界。
- **你已经花了一个多小时。** 对于大多数 bug，如果一小时的调查还没有产生清晰的假设，问题已经足够复杂，应该由工程团队接手。超过此点你的时间收益递减。

**尊重工程时间的交接协议。** 交接时，清晰地沟通：

"我花了 30 分钟调查。这是我发现的：[摘要]。我的假设是 [假设]。我停下来是因为 [原因，如需要生产数据或代码太复杂]。你能从这里接手吗？"

这告诉工程团队你做了什么、你怎么想以及从哪里开始。他们不会重复你的工作或想知道你已经尝试了什么。

**递减回报曲线。** 调查的前 20 分钟产生了 80% 的上下文价值。你学到了相关文件、形成了假设并识别了缺口。接下来的 40 分钟产生大约 15% 的额外价值：确认细节、排除替代方案、记录边缘情况。超过一小时，你可能是在没有进展地迭代。

**图 4.5：调查价值曲线** ——用于知道何时停止调查并交接给工程团队（20 分钟规则）。曲线在前 20 分钟陡峭上升，捕获 80% 的上下文价值，然后显著趋于平坦。20 分钟处有一条粗体垂直线标注"交接点"，附有标注："在此停止——工程师从这里接手。"绿色区域（0-20 分钟）显示高 PM 价值；黄色区域（20-40 分钟）显示中等收益；灰色区域（超过 40 分钟）表示递减回报。

- **对于直接明了的 bug**：15-20 分钟调查，然后交接。
- **对于复杂 bug**：最多 30-45 分钟，然后带着你尝试了什么的清晰文档交接。
- **对于关键/紧急 bug**：10 分钟收集初始上下文，然后与工程团队并行工作，而非顺序交接。

**何时完全跳过调查。** 有些问题不值得 PM 调查：
- 用户报告的安全漏洞：立即升级给安全团队
- 生产中断：工程团队处理事故响应，不是 PM
- 需要立即热修复的问题：调查延迟行动
- 复现清晰且影响明显的 bug：直接提交工单即可

当上下文收集节省的工程时间超过花费的 PM 时间时，调查增加价值。对于明显或紧急的问题，直接跳到交接。

你现在可以在不依赖工程团队进行初始分类的情况下调查 bug。你理解了如何将用户投诉翻译为技术查询、遵循结构化调查工作流、识别常见 bug 模式、撰写有用的 bug 报告以及知道何时停止。这种能力改变了你与工程团队的关系。你是调查的合作伙伴，而不仅仅是工单转发器。

# 第 5 章 市场与竞争分析

## 5.1 为什么 Claude Code 在研究方面胜过 Claude.ai

你的利益相关者要在周五前拿到一份竞争分析。你有三个选择。第一：花两天手动将信息汇编到 Google 文档中，下个季度就过时了。第二：用 Claude.ai 快速对话，产出不错但关闭标签页就消失了。第三：用 Claude Code 生成结构化、可版本化的产物，存放在你的代码仓库中，可以系统性地更新。

对于一次性问题（"竞品 X 的定价模式是什么？"），Claude.ai 就够了。对于需要持久化、可在 PRD 中引用、每季度更新、跨团队共享的研究，Claude Code 是正确的工具。区别不在于能力，而在于持久性和可重复性。

**为什么用 Claude Code 而非 Claude.ai 做研究？**

- **持久性。** 研究输出成为仓库中的文件。它们在会话关闭后仍存在，可以通过文件路径在文档中引用，并与团队的现有工作流集成。你今天生成的竞争矩阵，下个月 VP 要求再看时作为 `research/competitive-analysis-2026-01.md` 就在那里。
- **文件输出。** Claude Code 撰写结构化文档：markdown 表格、比较矩阵、格式一致的列表。Claude.ai 给你需要复制粘贴和重新格式化的文本。Claude Code 产出可直接提交的产物。
- **可重复性。** 你构建一次研究方法论（提示词、输出格式、分析框架），然后持续一致地执行。季度竞争审查使用相同的流程，使结果跨时间可比较。你不需要每个周期都重新发明研究结构。
- **可版本化。** Git 中的研究产物追踪变更。你看到 Q1 和 Q2 竞争分析之间的变化。你可以 diff 定价比较来发现趋势。这对向利益相关者展示"竞争格局如何变化"很重要。

**网页端 Claude 何时足够**：探索性研究，你还在弄清该问什么问题。快速事实核查。头脑风暴研究方向。任何对话比产物生成更有价值的场景。Claude.ai 在迭代对话方面表现出色。用它来思考，用 Claude Code 来记录。

**研究工具选择**：Claude.ai 用于探索性对话 vs. Claude Code 用于结构化、版本化的产物。

生成竞争分析的会话花费 $0.30-0.75，耗时 15-25 分钟。对比两天的体力劳动或过时研究导致糟糕产品决策的成本。如果你每个季度不止做一次市场研究，ROI 是立竿见影的。

## 5.2 构建可更新的竞争情报

你需要知道你在与谁竞争、他们提供什么以及你的差异化何在。手动研究涉及浏览竞品网站、做笔记、试图将所有东西保存在一个不一致且不完整的电子表格中。Claude Code 将其系统化。

**设置竞争分析工作区。** 为研究创建目录结构：

```bash
mkdir -p research/competitors
cd research/competitors
```

这使研究与代码分离，易于查找，可以按日期或主题组织。结构可能是：

```
research/
├── competitors/

```

在仓库根目录以文件写入权限启动 Claude Code：

> **提示词模板：结构化竞品档案。** 有用的竞争分析的关键是一致性。每个竞品档案应以相同格式回答相同问题。定义你的结构，然后反复执行。

> **提示词：竞品档案**
>
> 使用网络搜索研究 [竞品名称]，在 `research/competitors/[competitor-name]-profile.md` 创建结构化档案。
>
> 包含以下部分：
> - **概览**：一段话描述他们做什么、服务谁、公司阶段
> - **产品服务**：核心产品和关键功能（列表）
> - **定价模式**：如何收费、有哪些层级、大致价格
> - **目标市场**：卖给谁（公司规模、行业、用户角色）
> - **关键差异化**：他们在定位中强调什么（3-5 条）
> - **优势**：相对于我们，他们强在哪里
> - **劣势**：他们在哪里弱或有缺口（3-5 条）
> - **近期动态**：产品发布、融资、合作（过去 6 个月）
> - **最后更新**：[今天日期]
>
> 使用网络搜索收集当前信息。保持每部分简洁（这是参考文档，不是深度分析）。

Claude Code 搜索网络获取信息，综合发现结果，撰写结构化 markdown 文件。输出生成需要 5-10 分钟，花费 $0.15-0.35，取决于网络搜索量。

你得到什么：一份一致、结构化的档案可供你参考。每个竞品都有相同的部分，使对比直观。当 CEO 问"竞品 X 与我们有什么不同？"时，你打开他们的档案，阅读差异化部分。

**输出格式：比较矩阵和功能表。** 单独的档案很有用，但利益相关者想要对比。一旦有了 3-5 个竞品档案，生成一个矩阵：

> **提示词：竞争矩阵**
>
> 基于 `research/competitors/` 中的竞品档案，在 `research/competitors/competitive-matrix-2026-01.md` 创建一个比较矩阵。
>
> 创建一个 markdown 表格比较：
> - 核心功能（行：认证、权限、集成、移动端支持、API 访问、报表等）
> - 竞品（列：我们、竞品 A、竞品 B、竞品 C、竞品 D）
> - 值：✓（完全支持）、~（部分/有限）、x（不可用）、?（未知）
>
> 包含一个定价比较表：
> - 行：免费层、入门级、专业级、企业级
> - 列：相同竞品
> - 值：价格/月 或 "联系销售" 或 "N/A"
>
> 在每个表格后添加简短分析（2-3 句话），突出关键模式。

Claude Code 读取你已有的竞品档案，提取相关信息，格式化为可快速浏览的表格。矩阵展示缺口：竞品有你没有的功能、你定价不具竞争力的层级、你应该强调的优势。

**将结果存储为可版本化的产物。** 将这些文件提交到 git：

```bash
git add research/competitors/
git commit -m "Add Q1 2026 competitive analysis"
```

现在你的研究有版本控制了。三个月后，更新档案并生成标有 Q2 的新矩阵。diff 文件查看变化：

```bash
git diff competitive-matrix-2026-01.md competitive-matrix-2026-04.md
```

你看到竞品 A 新增了移动功能，竞品 B 提价了，一个新竞品进入了市场。这个历史视图展示格局演变，而不仅仅是当前状态。

**竞争格局映射的成本和时间。** 初始设置（5 个竞品档案 + 矩阵）：45-60 分钟，80,000-150,000 token，$0.40-0.75。季度更新（刷新档案 + 重新生成矩阵）：20-30 分钟，40,000-70,000 token，$0.20-0.35。对比每个周期 8-12 小时的手动研究。

**此方法何时失效**：公开信息稀缺的利基行业。定价不透明的竞品（一切都要"联系销售"）。信息每周就过时的快速变化市场。在这些情况下，Claude Code 仍然结构化你所知的内容，但你需要用销售电话、分析师报告或客户访谈来补充。

## 5.3 无需电子表格追踪竞品定价

定价情报需要系统性收集和结构化比较。你的竞品收费 $49/月，但包含什么功能？什么限制？与你的 $59/月层级相比如何？手动研究捕获了定价但遗漏了上下文。Claude Code 两者兼得。

**系统性地收集定价情报。** 定价页面会变。促销会上线。折扣会出现。你需要一个可以重复的快照方法论，以便随时间追踪变化。

> **提示词：定价深度分析**
>
> 使用网络搜索研究 [竞品名称] 当前的定价模式，在 `research/pricing/[competitor-name]-pricing-2026-01.md` 创建详细分解。
>
> 包括：
> - **定价层级**：名称、月价格、年价格（如不同）
> - **每层功能限制**：用户数、存储、API 调用、项目数等
> - **关键功能门槛**：哪些功能被锁定在更高层级？
> - **附加项**：可选额外功能及其定价
> - **折扣**：年付折扣百分比、批量折扣、非营利/教育定价
> - **免费层**：包含什么、有什么限制
> - **企业层**：定价公开还是"联系销售"？包含什么？
> - **计费模式**：按用户、按功能、按用量、固定费率？
>
> 使用网络搜索收集当前信息。如果某事没有公开记录，标注为"未披露"。
> 在顶部添加"最后更新"日期。

Claude Code 搜索定价页面，提取细节，一致地结构化。你得到一份远比"他们收费 $49/月"详细的定价分解。你知道那 $49 包含什么、不包含什么，以及与其他层级相比如何。

**提示词模板：定价模式比较。** 一旦有了单个竞品定价文档，比较它们：

> **提示词：定价比较表**
>
> 基于 `research/pricing/` 中的定价文件，在 `research/pricing/pricing-comparison-2026-01.md` 创建比较表。
>
> 创建表格：
> - 行：定价层级（免费、入门/基础、专业/标准、企业）
> - 列：我们、竞品 A、竞品 B、竞品 C
> - 值：每用户每月价格（或如果不是按用户计费，则为月度总价）
>
> 在定价表下方，创建功能限制比较：
> - 行：关键限制因素（用户数、存储、API 调用、项目数）
> - 相同列
> - 值：实际限制（如："10 用户"、"100GB"、"1000/天"）
>
> 添加分析：
> - 每个层级哪个竞品最便宜/最贵？
> - 我们在哪里定价有竞争力 vs. 溢价 vs. 折扣？
> - 每个价格点的价值主张是什么？

这揭示了定价策略差异。也许竞品 A 标价更便宜但按用户算更贵。也许竞品 B 在相同价格提供更慷慨的限制，表明你应该重新考虑你的上限。也许你是唯一有真正免费层的，这是你没有足够强调的竞争优势。

**通过信息传达审计分析定位。** 定价不只是数字，而是竞品如何定位价值。他们强调什么？他们声称解决什么问题？他们引用什么客户类型？

> **提示词：定位分析**
>
> 使用网络搜索研究 [竞品名称] 的网站，在 `research/competitors/[competitor-name]-positioning.md` 分析其定位。
>
> 从其首页、产品页面和营销内容中提取和分析：
> - **主标题**：他们的主要价值主张是什么？
> - **关键信息主题**：什么概念反复出现？（如："快速"、"安全"、"简单"、"企业级"）
> - **引用的客户类型**：他们说他们服务谁？
> - **问题陈述**：他们解决什么客户痛点？
> - **差异化声明**：什么让他们"不同"或"更好"？
> - **社会证明**：他们展示什么案例研究、Logo、客户评价？
> - **行动号召**：他们希望访客做什么？（试用、演示、联系销售）
>
> 提供 2-3 句话总结其定位策略。

你了解到竞品 A 定位为"面向受监管行业的企业级安全"，而竞品 B 定位为"最快上手的方式（无需设置）"。这些是不同的价值主张，吸引不同的客户。你的定位需要要么与两者区分开来，要么瞄准一个服务不足的细分市场。

**用标注日期的产物追踪变化。** 定位会变化。竞品在进入新市场时、融资轮次后或应对竞争时会重新定位。追踪这些揭示了策略。

将定位分析按日期保存：`competitor-a-positioning-2026-01.md`、`competitor-a-positioning-2026-04.md`。当你比较版本时，你看到信息传达的演变。也许竞品 A 不再提"小团队"，开始强调"企业规模"。他们正在向高端市场移动。这为你自己的定位决策提供信息。

**定价分析的时间和成本**：一个竞品的定价深度分析：10-15 分钟，20,000-35,000 token，$0.10-0.18。5 个竞品的定价比较表：10 分钟，15,000-25,000 token，$0.08-0.13。每个竞品的定位分析：15-20 分钟，25,000-40,000 token，$0.13-0.20。

**何时失效**：复杂定制化定价的竞品，公开信息不反映实际成交。不透明的企业定价，没有列出的数字。在这些情况下，用销售情报补充 Claude Code 研究：潜在客户告诉你的竞品报价、赢单/丢单分析、与前竞品客户的对话。

## 5.4 创建经得起推敲的市场估算

利益相关者想要数字。这个市场有多大？多少潜在客户？我们的可触达机会是什么？市场估算需要假设、数据来源和结构化的推理。Claude Code 不会魔法般地产生准确的数字，而是结构化你的方法论，使你的假设可见且经得起质疑。

**使用 Claude Code 结构化 TAM/SAM/SOM 分析。** 总可触达市场（TAM）、可服务可触达市场（SAM）、可服务可获得市场（SOM）：这是经典的风投框架。你需要估算这些、记录你的推理并使其可被审查。

> **提示词：市场估算框架**
>
> 帮我使用 TAM/SAM/SOM 框架估算 [产品/类别] 的市场规模。创建 `research/market-sizing/market-size-2026.md`。
>
> 对每个指标提供：
> - **定义**：这个指标对我们的产品意味着什么
> - **计算方法**：我们将如何估算它
> - **数据来源**：数字从哪来（行业报告、人口普查数据、代理指标）
> - **假设**：我们假设什么是真实的
> - **估算范围**：低/中/高估算
> - **推理**：为什么这个计算是合理的
>
> 结构：
> 1. **TAM** — 总可触达市场（如果每个可能使用此产品的人都使用了）
> 2. **SAM** — 可服务可触达市场（我们能服务的现实子集）
> 3. **SOM** — 可服务可获得市场（我们在 1-3 年内可以捕获的）
>
> 展示你的工作过程。使假设明确且保守。

Claude Code 帮助你理清逻辑：

```
## 总可触达市场 (TAM)
**定义**: 北美所有拥有 50-5000 名员工、需要 [产品类别] 的公司。
**计算方法**:
- 北美拥有 50-5000 名员工的公司总数：~850,000（美国人口普查局，2024 年）
- 需要 [产品类别] 的比例：~30%（行业报告显示类似工具的渗透率）
- 公司 TAM：850,000 × 30% = 255,000 家公司
**收入 TAM**:
- 假设平均交易规模为 $5,000/年
- 255,000 家公司 × $5,000 = $12.75 亿
**假设**:
- 公司规模范围捕获了目标市场（排除了过小和过大的企业）
- 30% 的需求基于类似产品采纳率
- $5,000/年是现实的平均值（SMB 和中等市场的混合）
**估算范围**:
- 低：$8.5 亿（20% 需求，$4K 均值）
- 中：$13 亿（30% 需求，$5K 均值）
- 高：$21 亿（40% 需求，$6K 均值）
```

你不是把假设藏在电子表格里，而是文档化且可被审查。当你的 CFO 质疑 30% 渗透率时，你可以明确讨论并更新模型。

**提示词模板：带假设的市场估算。** 关键是记录方法论，而不是产生单一的"魔法数字"。

> **提示词：市场估算方法论审查**
>
> 审查 `research/market-sizing/market-size-2026.md` 中我的市场估算假设。对每个假设评估：
> - 数据来源可信吗？
> - 逻辑合理吗？
> - 有没有明显的缺口或有问题的跳跃？
> - 什么敏感性分析会有帮助？（如果这个假设偏差 50% 会怎样？）
>
> 提供建设性反馈以加强分析。

Claude Code 挑战你的推理。它可能指出："30% 渗透率假设没有来源，仅描述为'行业报告显示'。你能引用具体报告或提供代理计算吗？"或者："计算假设所有目标公司花费相同金额，但 SMB vs. 中等市场定价可能差异显著。考虑分段计算。"

这种反馈帮助你产出经得起推敲的市场估算，而不只是听起来不错的数字。

**细分框架及其应用。** 市场不是铁板一块。你有按公司规模、行业、地域或用例划分的细分市场。细分显示该聚焦哪里。

> **提示词：市场细分**
>
> 按 [公司规模 / 行业 / 地域 / 用例] 细分 [产品类别] 市场。在 `research/market-sizing/segmentation-2026.md` 创建细分分析。
>
> 对每个细分估算：
> - **规模**：多少潜在客户？
> - **特征**：什么定义了这个细分？
> - **需求**：他们最关心什么？
> - **付费意愿**：价格敏感度（高/中/低）
> - **竞争**：还有谁瞄准这个细分？
> - **我们的适配度**：我们今天服务这个细分的程度如何？（强/中/弱/无）
>
> 按吸引力排序：规模 × 付费意愿 × 我们的适配度。

你得到一份有清晰推理的优先排序细分列表。也许"中等市场医疗"得分高，因为它大、愿意为你拥有的合规功能付高价、且被聚焦于 SMB 或企业的竞品服务不足。这为市场进入策略提供信息。

**为利益相关者审查记录方法论。** 市场估算文档成为参考。你的董事会演示引用它们。你的销售团队用它们理解目标市场。记录方法论使数字可信。

在你的市场估算文档中加入"如何更新此分析"部分：

```
## 如何更新此分析
刷新这些估算：
1. 查看美国人口普查局数据中更新的公司数量
2. 查看 [具体行业报告] 中更新的渗透率
3. 根据实际销售数据验证平均交易规模
4. 用更新的输入重新运行计算
5. 更新"最后审查"日期
**审查节奏**：每年一次，或进入新市场时。
```

这使其他人（或将来的你）可以在不重新发明方法论的情况下更新分析。

**时间和成本**：初始市场估算分析：30-45 分钟，50,000-80,000 token，$0.25-0.40。细分分析：20-30 分钟，30,000-50,000 token，$0.15-0.25。用新数据更新：15-20 分钟，20,000-35,000 token，$0.10-0.18。

**局限性**：Claude Code 没有专有市场数据。它使用公开可用的信息，这意味着估算可能较粗糙。对于高风险决策如融资或并购，用付费分析师报告、调查或顾问来补充。对于产品策略和内部规划，Claude Code 的结构化方法已经足够。

## 5.5 在行业趋势中分离信号与噪音

行业趋势塑造产品策略。AI 能力在进化。隐私法规在收紧。客户期望在转变。你需要追踪趋势、将炒作与信号分开、综合对你的路线图的影响。这是受益于 Claude Code 综合和文档能力的研究工作。

**从多个来源聚合趋势数据。** 趋势不是单一文章，而是跨来源的模式。你阅读分析师报告、技术博客、客户对话、会议演讲。将其综合为连贯的趋势简报非常耗时。Claude Code 将其系统化。

创建趋势工作区：

```bash
mkdir -p research/trends
```

将来源（粘贴文章文本、复制报告部分、总结对话）收集到单个文件中：

```
research/trends/
├── ai-tooling-gartner-2026.txt
├── privacy-regulations-summary.txt
├── customer-feedback-themes.txt
└── saas-trends-2026.txt
```

然后让 Claude Code 综合：

> **提示词：趋势综合**
>
> 审查 `research/trends/` 中的来源文件，将关键主题综合为 `research/trends/trend-brief-2026-01.md`。
>
> 对每个识别出的主要趋势：
> - **趋势名称**：简洁标签
> - **描述**：正在发生什么变化（2-3 句话）
> - **证据**：什么来源支持此观点？（引用具体文件）
> - **时间**：这正在、即将还是将来发生？（1 年 / 2-3 年 / 3+ 年）
> - **对我们的相关性**：这如何影响我们的产品？（高 / 中 / 低）
> - **影响**：我们应该考虑做什么？（2-3 条）
>
> 按相关性排序。限制在前 5-7 个趋势，避免简报过于庞杂。

Claude Code 读取你的来源文件，识别反复出现的主题，产出结构化的趋势简报。你得到的是综合，而不仅仅是聚合。

**提示词模板：行业趋势综合。** 关键是将趋势与行动连接起来。

示例输出：

```
## 趋势：AI 驱动的生产力功能正在成为基本门槛
**描述**: 客户越来越期待产品中内置 AI 辅助：自动完成、建议、智能默认值。18 个月前还是差异化优势的功能，正在成为 SaaS 工具中预期的基线功能。
**证据**:
- Gartner 报告（ai-tooling-gartner-2026.txt）指出 67% 的 B2B 买家期望 AI 功能
- 客户反馈（customer-feedback-themes.txt）包含 12 处提到"为什么这不建议..."或"竞品有 AI 做这个"
- SaaS 趋势报告（saas-trends-2026.txt）将 AI 功能列为前 3 投资领域
**时间**: 正在发生。客户今天已在基于 AI 能力比较产品。
**对我们的相关性**: 高。我们 AI 功能很少，而竞品正在推出。
**影响**:
- 评估哪些工作流最受益于 AI 辅助
- 优先考虑改善核心工作流的 AI 功能，而非新奇功能
- 考虑 AI 能力的合作 vs. 自建
- 监控可能与 AI 相关的"缺失功能"导致的客户流失
```

这在来源之间连接各个点，使趋势可操作。你不是仅仅报告"AI 很热"，而是在展示证据、时间线和对产品具体的影响。

**分离趋势报告中的信号与噪音。** 不是每个趋势都重要。有些是炒作，有些是真实但与你的市场无关，有些重要但在遥远的未来。Claude Code 帮助你过滤。

> **提示词：趋势验证**
>
> 审查 `research/trends/trend-brief-2026-01.md` 中的趋势。对每个趋势评估：
> - **信号 vs. 噪音**：这有多个可信来源支持，还是只是一篇文章的猜测？
> - **相关性过滤**：这影响我们的目标客户细分，还是其他市场？
> - **紧迫性**：我们需要本季度、今年行动，还是只需监控？
>
> 标记任何看似没有实质内容的炒作趋势。建议哪些趋势值得立即纳入路线图考虑 vs. 长期监控。

这防止追逐每个光鲜的趋势。你得到现实检验。"'万物区块链'趋势出现在一个来源中，没有客户验证。建议降级。'隐私优先架构'趋势出现在法规分析、客户反馈和竞争定位中。这是信号。建议路线图审查。"

**创建可操作的趋势简报。** 输出应该推动决策，而不是躺在文件夹里。结构化趋势简报以供利益相关者消费：

> **提示词：高管趋势摘要**
>
> 基于 `research/trends/trend-brief-2026-01.md`，在 `research/trends/trend-summary-exec-2026-01.md` 创建一页高管摘要。
>
> 格式：
> - **我们必须响应的前 3 个趋势**：简短描述 + 推荐的具体行动
> - **需要监控的趋势**：2-3 个新兴但尚不紧迫的趋势
> - **我们忽略的趋势及原因**：1-2 个不适用于我们的趋势
>
> 保持可快速浏览。高管应能在 3 分钟内阅读并理解影响。

这成为你策略审查中的一页幻灯片、路线图规划中的参考或执行团队的简报。它以行动为导向，而不仅仅是信息性的。

**时间和成本**：将 5-8 个来源文档综合为趋势简报：20-30 分钟，40,000-70,000 token，$0.20-0.35。从趋势简报创建高管摘要：10 分钟，15,000-25,000 token，$0.08-0.13。趋势分析周期总计：30-40 分钟，$0.30-0.50。

**何时刷新**：快速变化行业每季度，较慢市场每年。重大行业事件发生时临时更新（新法规、主要竞品动作、技术突破）。

**局限性**：Claude Code 综合你给它的东西。如果你的来源质量低或偏向某个视角，综合也会如此。垃圾进，垃圾出。将 Claude Code 的综合与多样化的可信来源结合：分析师报告、客户研究、行业专家、竞争情报。

# 第 6 章 客户反馈与 UX 研究综合

## 6.1 反馈综合的挑战

你手上有上个季度 847 张支持工单、200+ 条 NPS 回复、15 份客户访谈记录，还有一个用户每天发布功能请求的 Slack 频道。你的 VP 想要洞察。什么主题浮现了？我们应该优先考虑什么？客户实际上在告诉我们什么？

传统方法：通读一切，高亮看起来重要的内容，创建带分类的电子表格，试图记住三小时前读到的内容，产出一份 60% 是你清晰记得、40% 是因为最近或戏剧性而留在脑海中的总结。这需要数天时间，并遗漏分散来源中出现的模式。

你需要**系统化综合**：一致的分类、频率分析、主题之间的联系、每个洞察的支持证据。大规模的手工综合不可靠。你的记忆过滤机制偏好近期性和显著性，而非真实的频率或重要性。你在中途形成更好理解时分类会不一致。你遗漏埋在大量数据中的弱信号。

Claude Code 将反馈综合系统化。它读取你所有数据，应用一致的分类，追踪频率，识别共现主题，并产出带支持引用的结构化报告。同样的分析方法论反复执行。季度 NPS 分析每个季度使用相同流程，使结果跨时间可比较。

**三个核心问题：**

- **数量问题**：太多反馈，不够多的模式。随着产品增长，反馈量超出人工分析能力。Claude Code 详尽处理数据，而非选择性。它读取全部 847 张工单，而非抽样。
- **格式问题**：数据分散在工具和格式中。反馈存在于 Zendesk、Intercom、Google Forms、邮件、Slack、销售电话笔记、Twitter 提及。Claude Code 原生处理多种文件格式：Zendesk 的 CSV、Intercom 的 JSON、访谈记录的纯文本、会议笔记的 markdown。
- **优先级问题**：什么最重要？原始反馈不会自带优先级分数。Claude Code 综合揭示模式：15% 的反馈提到导出慢，可能只是一条关键路径问题，客户礼貌地提到一次，而非五个用户反复大声要求的功能请求。

Claude Code 以每次综合会话 $0.40-0.90 的成本解决这些问题，而手动需要数天。

## 6.2 准备好数据以供分析

在综合之前，你需要可分析的数据。这涉及从工具导出、合并为统一格式以及清洗数据。

**导出：从工具中提取反馈数据。** 从 Zendesk 导出工单为 CSV、从 Intercom 导出对话为 JSON、从 Typeform/Google Forms 导出调查结果为 CSV。将访谈记录保存为纯文本 markdown 文件。将 Slack 消息复制为文本文件。

将这些文件放在 `data/feedback/` 目录中。Claude Code 原生处理 CSV、JSON 和 markdown 文件。不需要将它们转换为单一格式。

**文件格式**：CSV（工单、调查）、JSON（API 导出、Intercom）、Markdown/纯文本（访谈记录、笔记、Slack 消息）。

**数据清洗：删除可识别信息（PII）。** Claude Code 不会自动清洗 PII。在将数据传递给 Claude Code 之前，使用电子表格中的查找替换或在 Claude Code 中要求其跳过特定字段来删除姓名、电子邮件地址和可识别信息。

创建 `data/feedback/README.md` 解释每个文件包含什么、它来自哪里以及已知的数据质量问题。Claude Code 在开始分析时会读取此文件以获得上下文。

**图 6.1：反馈综合工作流** ——用于将原始反馈转化为结构化洞察。五个步骤：1) 从工具导出，2) 合并到 `data/feedback/` 目录，3) 用 Claude Code 进行初始主题分析，4) 审查和完善主题，5) 产出报告和可操作工单。

## 6.3 在数百条回复中寻找模式

Claude Code 最强大的反馈分析能力是跨大型数据集的**主题分析**：识别反复出现的主题、按频率排序并附支持证据。

> **提示词：主题分析**
>
> 分析 `data/feedback/` 中的所有客户反馈。识别关键主题和模式。对每个主题提供：主题名称、描述（1-2 句）、频率（大致百分比）、严重性（高/中/低）、代表性引用（2-3 条原话）、受影响的用户群组（如适用）。
>
> 如果数据包含多种来源（工单、调查、访谈），请指出主题在不同来源间是稳定的还是因渠道而异。输出结果保存到 `analysis/themes-YYYY-MM-DD.md`。

**输出格式：附支持引用的主题报告。** 结果包括一个按频率排序的主题表格、每个主题附带引用和分析的详细章节，以及建议的后续步骤。对于 500 条反馈的分析：10-20 分钟，50,000-100,000 token，$0.25-0.50。

**跨时间趋势追踪。** 定期运行相同分析（月度或季度），比较主题分布变化。"上个月 '慢导出' 是第 3 主题（12% 提及），这个月是第 1 主题（22% 提及）。情况在恶化。"

## 6.4 从访谈记录中提取洞察

访谈提供深度、上下文和未过滤的用户语言。但它们是有噪音的长文档。Claude Code 系统化提取。

**数据准备：** 将访谈记录以纯文本保存到 `data/interviews/`。每条记录以标识符（如 `interview-01-lead-pm.md`）标注。添加 `data/interviews/README.md` 列出每条记录的内容和受访者背景。

> **提示词：访谈综合**
>
> 分析 `data/interviews/` 中的所有访谈记录。提取：跨访谈的关键主题（附带频率和引用的模式）、痛点（用户挣扎的具体任务或工作流）、愿望（用户表达的未满足需求）、意外发现（让你惊讶的引用或模式）。输出保存到 `analysis/interview-synthesis-YYYY-MM-DD.md`。

**跨访谈模式。** 你发现 12 名受访者中有 9 名在无提示的情况下自发提到相同的导出工作流问题。4 名提到"我永远不知道我是否拥有最新数据"。这是信号。

**旅程映射和工作流分析。** 要求 Claude Code 从其分析中生成旅程地图或工作流可视化。

时间：分析 10-15 条记录，15-25 分钟，40,000-80,000 token，$0.20-0.40。

## 6.5 从洞察到行动

综合产生洞察。你需要将这些转化为工程团队可以执行的东西。

**研究方法到产物的流水线**（图 6.4）：原始反馈 → 综合 → 主题 → 用户故事 → 路线图工单。每个阶段提炼并结构化输出，供下游消费。

> **提示词：洞察转行动**
>
> 从 `analysis/themes-YYYY-MM-DD.md` 的反馈综合中，创建可操作用户故事和工单。对每个主题：对应的用户故事（"作为 [用户]，我想 [目标]，以便 [收益]"）、建议的优先级（基于频率 × 严重性 × 战略对齐）、可能需要调查的代码区域（如适用）、建议的下一步（工单/调查/研究）。

**优先级框架。** Claude Code 可以应用 RICE（触达 × 影响 × 信心 ÷ 努力）、价值 vs. 努力矩阵或自定义优先级框架：

> - 高价值 + 低努力 = 立即做
> - 高价值 + 高努力 = 规划项目
> - 低价值 + 低努力 = 如有时间做
> - 低价值 + 高努力 = 不做

Claude Code 生成草稿工单，你可以复制到 Jira 或 Linear。**时间**：映射主题到行动通常需要 15-20 分钟，30,000-50,000 token，$0.15-0.25。

# 第 8 章 技能剖析——结构与设计模式

## 8.1 从临时提示词到可重复工作流

你已经执行了反馈综合、竞争分析、bug 调查文档。每次你把提示词粘贴进 Claude Code 时，你都在重复流程结构——每次凭记忆。技能（Skill）消除了这种重复。

技能是存储在 `.claude/skills/` 中的文档化工作流，Claude Code 按需执行。定义一次流程，每次以一致的方式运行。输入变化，流程保持稳定。

## 8.2 什么是技能？

技能是一个 Markdown 文件（`SKILL.md`），它告诉 Claude Code 如何执行特定类型的任务。它包含描述（技能做什么）、输入（需要什么数据/文件）、流程步骤、输出格式和质量标准。

技能可以存放在两个位置：
- **项目**：`.claude/skills/`（团队共享，提交到 git）
- **个人**：`~/.claude/skills/`（跨所有项目跟随你，用于通用 PM 工作流）

## 8.3 技能组件

一个技能文件包含 frontmatter（YAML）和正文（Markdown 指令）。

```yaml
---
name: feedback-synthesis
description: Synthesize customer feedback from CSV/JSON into a theme report with supporting quotes.
---
```

正文定义 Claude Code 执行的步骤。为什么用 YAML frontmatter？它允许 Claude Code 在不读取整个文件的情况下匹配技能，实现自动调用。

## 8.4 SKILL.md 文件的详细构成

**Frontmatter 字段：** `name`（必需：唯一标识符，kebab-case）、`description`（必需：单行摘要——当用户请求匹配该描述时，Claude Code 可自动调用此技能）、`arguments`（可选：技能接受的命名参数，如 `chapter` 或 `source`）。

**正文部分：**
1. **目的声明**：技能完成什么任务
2. **预期输入部分**：Claude Code 应期望什么数据/文件
3. **流程步骤**：编号或项目符号的指令序列
4. **输出格式规范**：产生的 markdown 结构
5. **质量标准**：验证标准（"每个主题必须有 2+ 支持引用"）
6. **已知边缘情况或限制**：坦诚说明技能可能失败的地方

示例技能——`standup-notes`（从昨日 git 提交生成立会笔记）：

```markdown
---
name: standup-notes
description: Generate standup notes from yesterday's git commits.
---

## Steps
1. Run `git log --since="yesterday" --oneline` to get commits.
2. Group by author. For each author, list their commits with messages.
3. Format as bullet points: `- [Author]: [description] (commit hash)`
4. Save to `standup-notes/YYYY-MM-DD.md`.
```

**设计原则**：约束性（更具体的指令 = 更一致的输出）、全面性（包含边缘情况和错误处理）、自适应（使用条件逻辑处理变化："如果 X 存在，使用 X；否则问用户"）。

## 8.5 PM 技能的设计模式

**图 8.3：五种 PM 技能模式**

1. **转换（Transform）**：原始数据 → 结构化报告（例如：反馈 CSV → 主题报告）
2. **调查（Investigate）**：问题 → 带证据的答案（例如："这个 bug 的原因是什么？" → 文件引用 + 假设）
3. **生成（Generate）**：参数 → 工件（例如：功能名称 → PRD 草稿）
4. **审计（Audit）**：工件 → 检查结果（例如：PRD → 质量评分 + 缺失元素）
5. **协调（Orchestrate）**：目标 → 多步骤工作流（例如："准备发布" → 变更日志 + 工单验证 + 利益相关者邮件）

## 8.6 确定性与可重复性

技能的目标：相同的输入应产生基本相同的输出。这不意味着逐字相同（AI 具有固有的可变性），但结构应相同，主题应相似，措辞应稳定。

如果输出差异很大，你的技能需要更严格的约束。添加更多具体的指令。指定确切的输出格式。

## 8.7 技能组合

技能可以串联："运行 feedback-synthesis，然后将结果传递给 roadmap-impact 以创建优先排序工单。"Claude Code 逐步执行每个技能——第一个的输出成为第二个的输入。第 9 章涵盖构建完整的 PM 技能库。
# 第 9 章 构建你的 PM 技能库

## 9.1 入门套件：五项必备 PM 技能

你已构建了一个有效的反馈综合工作流，运行了三次，每次节省 45 分钟。现在是系统化的时机。

**五项必备技能：**

1. **feedback-synthesis**：将 CSV/JSON 反馈数据转化为附有支持引用的主题报告。输入：`data/feedback/` 中的反馈数据文件。输出：`analysis/themes-YYYY-MM-DD.md`。

2. **feature-investigation**：调查功能或 bug 并产出附有文件引用的报告。输入：自然语言问题。输出：带代码库证据的调查报告。

3. **competitive-profile**：为竞品创建结构化分析档案，以便为比较矩阵提供数据。输入：竞品名称。输出：`research/competitors/[name]-profile.md`。

4. **prd-generator**：从用户故事或功能概念创建带有代码库引用的 PRD 草稿。输入：功能描述 + 用户故事文件。输出：`docs/prds/[feature]-prd.md`。

5. **release-notes**：从 git 历史生成发布说明，格式化为面向用户的变更日志。输入：git 引用（标签、分支或日期范围）。输出：`docs/changelog/vX.Y.Z.md`。

## 9.2 逐步创建你的第一个技能

**步骤 1：选择可重复的任务。** 从你已经以临时方式执行过三次以上的工作流开始。

**步骤 2：记录当前流程。** 写下你实际做了什么步骤——确切的提示词、你提供的输入、你期望的输出格式。

**步骤 3：创建 SKILL.md 模板。**

```markdown
---
name: my-skill-name
description: 一句话描述此技能的功能及触发时机。
---

## 目的
[此技能完成什么]

## 输入
[预期什么文件/数据]

## 流程
1. [步骤 1]
2. [步骤 2]
3. [输出格式规范]

## 质量标准
- [每个输出必须满足的标准]
```

**步骤 4：使用真实输入测试。** 用实际数据运行它。输出是否符合质量标准？约束是否足够严格？提示词中是否有缺失步骤？

**步骤 5：基于结果迭代。** 第一次运行很少完美。根据输出调整指令。收紧模糊部分。添加在测试中发现的边缘情况。

**步骤 6：提交到 `.claude/skills/`。** 技能是代码仓库的一部分。团队可以在代码中看到它，版本控制追踪变更。

## 9.3 保持技能长期有效

技能会退化。API 变更，文件结构演变，团队约定变化。随着 Claude Code 本身的更新，一些指令可能过时。

**维护实践：**
- **定期运行。** 如果你一个季度没用某个技能，运行时预期需要调整。
- **用当前数据测试。** 反馈综合需要最近的文件。竞争分析需要当前的市场状况。
- **失败时更新。** 如果技能没有产出预期输出，之后立即更新——不要让它"下次再坏"。
- **版本化技能。** 它们是代码——提交消息应描述变更内容及原因。

## 9.4 使技能适应不同请求

**自动调用。** 当你的请求匹配技能的 `description` 字段时，Claude Code 会自动加载该技能。只需描述你想要什么："分析 data/q1-2026-feedback.csv 中的反馈并创建主题报告。"Claude Code 看到技能描述匹配并加载 feedback-synthesis。

**基于上下文的条件行为。** 高级技能使用条件逻辑：

```markdown
**如果 `.claude/feedback-categories.md` 存在**：
- 使用该文件中的预定义分类
**如果 `docs/personas/` 目录存在**：
- 按受影响画像对每个主题进行分段
```

这使得技能能够适应不同的项目环境而无需修改。

## 9.5 追踪技能是否为你节省时间

测量采用情况和影响："技能运行次数/月"、"每次运行的平均时间节省 vs. 手动执行"、"产出质量（一致吗？全面吗？）"。

如果技能持续提供价值，可以考虑将其推广为团队范围内的标准。如果某个技能被闲置，要么删除它，要么改进描述使其更易被发现。

# 第 10 章 MCP：连接到你的 PM 工具栈

## 10.1 MCP 如何消除导出-导入循环

模型上下文协议（Model Context Protocol，MCP）让 Claude Code 直接连接到外部工具——无需导出 CSV、上传到 Claude.ai 并复制粘贴回来。Claude Code 原生查询你的工具并就地更新。

**没有 MCP**：从 Jira 导出工单 → 保存为 CSV → 上传到 Claude Code → 分析 → 将结果复制粘贴回 Jira。

**有 MCP**：Claude Code 直接查询 Jira → 分析工作 → 用发现结果更新工单。

MCP 消除的是工具之间的手动交接——正是这些交接耗费动力并引入错误。

Claude Code 可以连接到各种工具：Jira（问题追踪器）、Slack（消息和频道）、GitHub/GitLab（仓库和 PR）、Figma（设计文件）、数据库（分析查询）、自定义 API。

## 10.2 配置你的第一个集成

MCP 配置存放在 `.claude/mcp.json`（项目级别）或 `~/.claude/mcp.json`（用户级别）。

**示例配置（Jira + Slack）：**

```json
{
  "mcpServers": {
    "jira": {
      "command": "npx",
      "args": ["-y", "@anthropic/jira-mcp-server"],
      "env": {
        "JIRA_URL": "https://your-company.atlassian.net",
        "JIRA_EMAIL": "you@company.com",
        "JIRA_API_TOKEN": "your-token"
      }
    },
    "slack": {
      "command": "npx",
      "args": ["-y", "@anthropic/slack-mcp-server"],
      "env": { "SLACK_BOT_TOKEN": "xoxb-..." }
    }
  }
}
```

MCP 服务器以独立进程运行。Claude Code 按需启动它们。

**权限范围**：MCP 服务器可以请求不同的权限级别——只读访问或读写访问。授予与你的信任水平匹配的最小权限。

## 10.3 在不离开 Claude Code 的情况下查询和更新 Jira

配置好 Jira MCP 后，Claude Code 可以：

- **查询工单**："显示当前 sprint 中所有阻塞的工单"、"提取分配给前端团队的所有 P1 bug"
- **创建工单**："为批量用户导入功能创建任务分解，给我本周冲刺中的四个任务"
- **更新工单**："用我在此对话中的发现更新 PM-1234 的描述，并添加包含文件引用的评论"
- **检查状态**："PM-1567 的状态是什么？谁在处理？"

**提示词模式**："根据我的 sprint 中的工单生成一份状态报告。按工单状态分组，并包含阻碍项。"

Claude Code 查询 Jira、按状态分组并格式化为报告——所有这些都在一个环境中完成。

## 10.4 自动拉取上下文和发布更新

MCP 的真正优势在于上下文收集和自动更新：

- **冲刺规划**："拉取当前 sprint 中所有工单的摘要及每个工单的分配情况。根据最近的 git 活动对其完成度进行评论。"
- **状态更新**："找到 PRD 中引用但尚未在 Jira 中创建工单的用户故事。为每个创建草稿工单，链接到 PRD。"
- **Slack 集成**："将状态更新发布到 #product-updates 频道，摘要发布说明中的关键变更。"

工作流：Claude Code 从 Jira 读取当前 sprint → 与 git 活动交叉引用 → 生成附有状态评论的冲刺摘要 → 发布到 Slack（如要求）。

**成本**：MCP 添加了工具调用，这增加了少量 token 开销。典型的 Jira/Slack 集成为会话成本增加不到 10%，但通过消除手动复制粘贴节省的时间远超此成本。

## 10.5 以编程方式提取设计规范

Figma MCP 让 Claude Code 访问设计文件：

> "从 [Figma 链接] 的设计文件中提取按钮样式和颜色令牌，并与 `src/styles/theme.js` 中定义的内容进行比较。标记任何差异。"

Claude Code 读取 Figma → 提取设计令牌 → 与代码库实现比较 → 生成差异报告。

## 10.6 无需编写 SQL 拉取指标

数据库 MCP 连接允许自然语言查询：

> "上个月有多少用户执行了导出操作？按计划层级细分。" → Claude Code 生成 SQL，运行它，并以表格形式呈现结果。

或："显示过去 3 个月按周划分的用户激活率趋势。"Claude Code 查询数据库 → 格式化为表格 → 可选地生成火花线图趋势描述。

**安全考虑**：数据库 MCP 应使用只读凭证。永远不要授予可能修改数据的 MCP 服务器写入权限，除非经过工程审查。MCP 集成是第 10 章中最强大的概念，但也最具风险。
# 第 11 章 用于专业任务的子智能体

## 11.1 子智能体如何实现并行工作

Claude Code 可以启动多个 Claude 实例（子智能体/subagent）并行工作或聚焦于特定子任务。作为 PM，你很少直接调用它们，但它们在复杂工作流中支撑了并行处理。

**顺序 vs. 并行：**
- 顺序：Claude Code 逐个处理三个竞品（A → B → C → 汇总）。时间：~15 分钟。
- 并行：Claude Code 为每个竞品启动一个子智能体，三者同时研究，然后汇总结果。时间：~5 分钟。

## 11.2 PM 常见工作流的四种模式

1. **分而治之研究**：将研究分割为独立的块。"研究竞品 A、B、C 的定价"启动三个并行子智能体，每个执行 `competitive-profile` 技能，结果汇总为比较矩阵。

2. **多维度分析**：从不同角度分析相同数据。"从以下角度分析这个反馈 CSV：按频率的主题、按用户群组的主题、随时间变化的主题趋势。"三个并行分析，一份综合报告。

3. **验证与审查**：独立验证发现结果。"我调查了一个 bug，假设 X。启动一个子智能体 adversarial（对抗性地）审查我的调查——我遗漏了什么？"一个独立的子智能体充当魔鬼代言人。

4. **工作量分配**：将大任务拆分为可管理的块。处理 2,000 条反馈？每个子智能体处理 500 条，然后合并结果。

## 11.3 有效协调多个智能体

**编排规则：**
- **独立任务 = 并行。** 任务间没有依赖关系？并行运行它们。
- **有依赖关系 = 顺序。** 任务 B 需要任务 A 的输出？按顺序运行。
- **清晰定义每个子智能体的范围。** 模糊的任务定义产生不一致的结果。
- **标准化输出格式。** 所有子智能体必须返回相同结构的数据以实现合并。

**合并结果**：并行执行后，所有子智能体的输出被馈入最终的综合步骤。协调器智能体读取每个子智能体的输出并生成统一报告。

## 11.4 限制与何时避免子智能体

**成本影响**：子智能体同时运行多个 Claude 实例。每个都消耗 token。并行执行 5 个子智能体可能花费 $2-5，而顺序执行相同的任务可能花费 $0.50-1.00。速度有其成本。

**何时避免：**
- **小任务**（< 5 分钟顺序执行）：并行化的开销不值得
- **紧密耦合任务**：如果一个子智能体的输出显著影响另一个的输入，按顺序执行它们
- **探索性工作**：在任务定义模糊时使用单个 Claude 实例进行迭代探索，只有在问题被清晰定义后才并行化

**实际经验法则**：如果顺序执行耗时超过 15 分钟，且任务可以清晰地分割成独立的部分，子智能体可能提供价值。

子智能体是 PM 的一个高级概念。先从单任务技能开始掌握（第 8-9 章），只有在对 Claude Code 的基本能力有信心后才探索并行工作流。
# 第 12 章 与工程团队协作

## 12.1 PM 与工程使用的重叠之处

PM 和工程师以不同方式使用 Claude Code，但有意义的协作存在重叠。工程师用它编写和调试代码。PM 用它调查和理解代码。重叠发生在**调查和上下文共享**上。

**情景**：PM 调查一个 bug → 产出附有文件引用和假设的调查报告 → 工程团队在相同的文件引用处开始他们的技术调试。调查的连续性保持上下文不丢失。

## 12.2 在不丢失上下文的情况下交接工作

**结构化的交接包：**
- PM 的调查摘要（第 4 章模板）
- 相关文件路径及行号引用
- 已测试并排除的假设
- 需要运行时验证的开放问题
- 建议的下一步

**在交接中包含你的 CLAUDE.md 上下文**：如果你在 CLAUDE.md 中构建了产品上下文（第 3.5 节），工程团队自动获得你已建立的相同制度知识。他们从上下文开始，而非空白状态。

**基于文件的交接**：因为所有内容都存在于仓库中（`docs/investigations/`、`analysis/`、`docs/bug-reports/`），工程团队可以在他们的 Claude Code 会话中打开这些文件。交接包是可被解析的工件，而非必须找到的聊天消息。

## 12.3 维护两种角色都能使用的上下文

**CLAUDE.md 作为共享制度知识：**
- PM 贡献：产品术语、用户旅程映射、常见 PM 问题、业务规则
- 工程团队贡献：架构决策记录、技术债务背景、部署说明
- 双方受益：一致的术语、共享的心智模型、更少的重复解释

**约定的共同所有权**：CLAUE.md ——像任何共享文档一样——需要共同维护。当 PM 了解到新的业务规则时，更新它。当工程团队进行架构变更时，更新它。将其视为团队运行手册。

## 12.4 构建服务整个团队的技能

**团队共享技能**：在 `.claude/skills/` 中的技能对拥有仓库访问权限的所有人可见。构建同时服务 PM 和工程用例的技能：

- `release-notes`：PM 用于利益相关者沟通，工程用于部署文档
- `bug-report`：PM 用于调查，工程使用相同的模板记录发现
- `competitive-analysis`：PM 拥有内容，工程贡献技术能力评估

**技能代码审查**：让工程团队审查你的技能。他们可以改进 git 命令、建议更好的文件路径模式并加强质量标准。这是尊重他们专业知识的低摩擦协作。

**协作的边界**：PM 不承诺代码变更。工程团队不拥有产品决策。Claude Code 作为共享工具，双方在不越界的情况下使用。当 PM 使用 Claude Code 进行调查时，他们行使了理解权，而非实现权。

---

当 PM 和工程团队都使用 Claude Code 时，它成为共享基础设施——一种调查、记录和决策的共同语言。上下文随工件保留，交接变得更干净，制度知识随仓库增长。
# 第 13 章 失败模式与边界

## 13.1 常见的 PM 失败模式

**过度自信于 AI 生成的解释。** Claude Code 阅读代码并提供解释，但它无法运行代码。其解释是高置信度假设，而非绝对真理。当准确性重要时（面向客户的沟通、路线图决策、安全评估），在行动前与工程团队验证。

**将 Claude Code 用于错误的工具。** 如果你在用 Claude Code 起草邮件、头脑风暴概念或获取快速答案，你正在错误的地方。Claude.ai 更适合这些。使用 Claude Code 做它设计用来做的事：基于文件的调查、工件生成、可重复工作流。

**建立过于复杂的技能。** 从简单开始。一个能做好一件事的 30 行技能比一个试图覆盖所有边缘情况但却失败的 300 行怪物更有价值。先构建最小可行技能，基于使用迭代。

**在数据准备上偷工减料。** Claude Code 合成你给它的东西。垃圾进，垃圾出。在要求合成之前，花时间清洗反馈数据、正确格式化文件并组织研究材料。

**忽视成本，然后感到震惊。** 作为 API 用户，长时间会话累积起来很快。每会话后检查 `/cost`。设置预算警报。如果成本过高，使用 `/compact`。作为订阅用户，不要忽视 token 消耗——高消耗意味着更慢的响应和更笨重的会话。

## 13.2 知道何时已提取了可用价值

**对调查的递减回报**（第 4.6 节进行了扩展）：前 20 分钟产生 80% 的价值。此后，除非你在探索全新的方面，否则继续迭代会支付递减回报。

**对技能构建的递减回报**：一旦技能对相同输入始终产生一致输出，额外的调整产生递减回报。停止调整，开始使用。

## 13.3 属于工程团队的任务

- **提交代码变更**。你可能调查并建议，但只有工程团队可以提交。
- **运行数据库迁移**。需要生产访问权限，超出 PM 范围。
- **做出架构决策**。Claude Code 可以告知，但不能决定。
- **安全审查**。需要只有工程师拥有的专业知识。
- **调试竞态条件**。需要运行时观察，而非静态代码分析。

Claude Code 赋予你理解权——而非修改或部署权。尊重该边界。

## 13.4 让 Claude Code 长期有效的习惯

1. **维护你的 CLAUDE.md。** 当领域知识变化时更新它。这是你的 AI 队友的入职文档，最好保持更新。
2. **使用 `/cost` 培养直觉。** 经过 10-15 个会话，你将凭直觉知道哪些类型的任务消耗 token。
3. **默认为 plan 模式。** 除非有意创建或修改文件，否则在只读模式下启动。安全默认设置防止事故。
4. **定期审查和刷新技能。** 每个季度，对每个技能进行一次测试运行。更新任何已失效的内容。
5. **记录你的发现。** 每次调查都应产出一个工件——一份文档，存在于仓库中。使制度知识显式化。
6. **在不确定时升级给工程团队。** 当调查遇到第 3.6 节中的任何边界时，交接并透明地说明你尝试了什么。
7. **使用适合每个任务的工具。** Claude.ai 用于思考，Claude Code 用于执行。两者都使用，有意识地使用。

Claude Code 是一款放大 PM 效能的工具——而非替代工程判断。有效使用它的 PM 会成为更好的合作伙伴：信息更充分，准备更充分，更尊重工程团队的时间。使用得当，它改变游戏规则。本书已向你展示了如何做到。

# 附录 B：SKILL.md 模板

创建新技能时复制此模板。完整示例见第 9 章，设计模式见第 8 章。

## 最小模板

```markdown
---
name: my-skill-name
description: 一句话描述此技能做什么，以及何时自动触发。
---

# [技能名称]

## 目的
[此技能完成什么——一段话，最多]

## 输入
- [预期此技能读取的文件或数据]

## 流程
1. [步骤 1：读取和验证输入]
2. [步骤 2：核心操作]
3. [步骤 3：格式化和保存输出]

## 输出格式
[生成的 markdown 结构]

## 质量标准
- [具体且可衡量的标准]
```

## 全面模板

```yaml
---
name: skill-name
description: 清晰描述技能功能。包含触发词，便于 Claude 在用户请求匹配时自动调用。例如："从 CSV 或 JSON 文件合成客户反馈为主题报告。当用户提到反馈分析、NPS 合成或支持工单主题时自动调用。"
---
```

正文包含：

1. **目的**：用户得到什么（而非技能内部做什么），一段话最多
2. **预期输入**：具体格式。"CSV 文件"太模糊；"文本列中包含反馈的 CSV"告诉 Claude 该找什么
3. **流程步骤**：按编号顺序执行。每一步可验证（"检查文件是否存在"、"识别相关列"）
4. **输出格式规范**：确切结构。"`# [报告标题]` → `## 主题：[名称]` → `- 频率：X%`"
5. **质量标准**：具体、可衡量。"每个主题必须有 2+ 条支持引用"、"表格必须使用一致的列标题"
6. **已知边缘情况或限制**："不处理图像数据"、"期望 CSV 使用 UTF-8 编码"

## 字段参考

| 字段 | 必需 | 说明 |
|------|------|------|
| `name` | 是 | 唯一标识符（kebab-case）。匹配文件名。 |
| `description` | 是 | 单行摘要。包含触发词用于自动调用匹配。 |
| 目的 | 是 | 领先用户获得什么的价值 |
| 流程 | 是 | 编号步骤。具体、可验证的指令 |
| 输入 | 推荐 | 期望的具体文件格式和结构 |
| 输出格式 | 推荐 | 输出 markdown 的确切结构 |
| 质量标准 | 推荐 | 可衡量的验证标准 |
| 边缘情况 | 可选 | 诚实说明限制 |

## 各部分指南

- **目的**：一段话最多。领先用户获得什么，而非技能内部做什么。
- **预期输入**：具体格式。"CSV 文件"太模糊；"文本列中包含反馈的 CSV"告诉 Claude 该找什么。
- **流程**：可验证的步骤。"读取输入文件"是模糊的；"检查 `data/feedback/` 中是否存在 CSV 文件"是可执行的。
- **输出规范**：明确指定结构。Claude 越是必须猜测，输出就越不一致。
- **质量标准**：可衡量的。"质量好"是模糊的；"每个主题必须有 2+ 条支持引用"是可验证的。

## 文件结构

```
.claude/skills/
├── SKILL.md            （可选：此目录的索引）
├── feedback-synthesis/
│   └── SKILL.md
├── feature-investigation/
│   └── SKILL.md
└── release-notes/
    └── SKILL.md
```

## 常见错误

- **描述缺少触发词**：Claude 将你的请求与技能描述进行匹配。如果你的描述不包含用户实际使用的词语，自动调用永远不会触发。
- **流程步骤不可验证**：如果 Claude 无法判断某个步骤是否完成，它将跳过或猜测。
- **输出格式太模糊**：如果不指定结构，每次运行会产生不同的格式。

- Pro tier: 5 workspaces, unlimited connectors, 1-year retention
- Enterprise: Custom limits, SSO, dedicated support
- Connector sync failures retry 3x before alerting user
- Pricing: Never calculated in frontend — all pricing logic in `src/services/pricing/`

## Team Conventions
- All customer-facing strings use i18n in `src/locales/`
- Database migrations require data team approval
- Feature flags managed in `src/config/features.js`

- Design system: https://figma.com/design-system
- Product strategy: [Google Docs link]
- Analytics dashboard: https://analytics.internal.acme.com

## Common PM Questions

- "Where is email content managed?" → `src/templates/email/`

## 逐部分指导

- **关于本产品（2-3 句话）**：它做什么？谁在使用它？在任何调查之前为 Claude Code 提供方向。
- **产品术语（5-10 个术语）**：定义在你的领域中具有特定含义或可能与常用术语混淆的词语。
- **用户旅程（3-5 个关键流程）**：将产品流程映射到代码位置。价值来自连接产品概念和实现。

- **外部资源（链接）**：Claude Code 无法访问但你需要参考的文档、设计、分析。
- **常见 PM 问题（快速参考）**：经常被调查的主题及回答指针。
