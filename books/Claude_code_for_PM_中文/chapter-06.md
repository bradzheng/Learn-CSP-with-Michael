# 第 6 章 客户反馈与 UX 研究综合

## 6.1 反馈综合的挑战

你手上有上个季度 847 张支持工单、200+ 条 NPS 回复、15 份客户访谈记录，还有一个用户每天发布功能请求的 Slack 频道。你的 VP 想要洞察。什么主题浮现了？我们应该优先考虑什么？客户实际上在告诉我们什么？

传统方法：通读一切，高亮看起来重要的内容，创建带分类的电子表格，试图记住三小时前读到的内容，产出一份 60% 是你清晰记得、40% 是因为最近或戏剧性而留在脑海中的总结。这需要数天时间，并遗漏分散来源中出现的模式。

你需要**系统化综合**：一致的分类、频率分析、主题之间的联系、每个洞察的支持证据。大规模的手工综合不可靠。你的记忆过滤机制偏好近期性和显著性，而非真实的频率或重要性。你在中途形成更好理解时分类会不一致。你遗漏埋在大量数据中的弱信号。

Claude Code 将反馈综合系统化。它读取你所有数据，应用一致的分类，追踪频率，识别共现主题，并产出带支持引用的结构化报告。同样的分析方法论反复执行。季度 NPS 分析每个季度使用相同流程，使结果跨时间可比较。

**三个核心问题：**

- **数量问题**：太多反馈，不够多的模式。随着产品增长，反馈量超出人工分析能力。Claude Code 详尽处理数据，而非选择性。它读取全部 847 张工单，而非抽样。
- **格式问题**：数据分散在工具和格式中。反馈存在于 Zendesk、Intercom、Google Forms、邮件、Slack、销售电话笔记、Twitter 提及。Claude Code 原生处理多种文件格式：Zendesk 的 CSV、Intercom 的 JSON、访谈记录的纯文本、会议笔记的 markdown。
- **优先级问题**：什么最重要？原始反馈不会自带优先级分数。Claude Code 综合揭示模式：15% 的反馈提到导出慢，可能只是一条关键路径问题，客户礼貌地提到一次，而非五个用户反复大声要求的功能请求。

Claude Code 以每次综合会话 $0.40-0.90 的成本解决这些问题，而手动需要数天。

## 6.2 准备好数据以供分析

在综合之前，你需要可分析的数据。这涉及从工具导出、合并为统一格式以及清洗数据。

**导出：从工具中提取反馈数据。** 从 Zendesk 导出工单为 CSV、从 Intercom 导出对话为 JSON、从 Typeform/Google Forms 导出调查结果为 CSV。将访谈记录保存为纯文本 markdown 文件。将 Slack 消息复制为文本文件。

将这些文件放在 `data/feedback/` 目录中。Claude Code 原生处理 CSV、JSON 和 markdown 文件。不需要将它们转换为单一格式。

**文件格式**：CSV（工单、调查）、JSON（API 导出、Intercom）、Markdown/纯文本（访谈记录、笔记、Slack 消息）。

**数据清洗：删除可识别信息（PII）。** Claude Code 不会自动清洗 PII。在将数据传递给 Claude Code 之前，使用电子表格中的查找替换或在 Claude Code 中要求其跳过特定字段来删除姓名、电子邮件地址和可识别信息。

创建 `data/feedback/README.md` 解释每个文件包含什么、它来自哪里以及已知的数据质量问题。Claude Code 在开始分析时会读取此文件以获得上下文。

**图 6.1：反馈综合工作流** ——用于将原始反馈转化为结构化洞察。五个步骤：1) 从工具导出，2) 合并到 `data/feedback/` 目录，3) 用 Claude Code 进行初始主题分析，4) 审查和完善主题，5) 产出报告和可操作工单。

## 6.3 在数百条回复中寻找模式

Claude Code 最强大的反馈分析能力是跨大型数据集的**主题分析**：识别反复出现的主题、按频率排序并附支持证据。

> **提示词：主题分析**
>
> 分析 `data/feedback/` 中的所有客户反馈。识别关键主题和模式。对每个主题提供：主题名称、描述（1-2 句）、频率（大致百分比）、严重性（高/中/低）、代表性引用（2-3 条原话）、受影响的用户群组（如适用）。
>
> 如果数据包含多种来源（工单、调查、访谈），请指出主题在不同来源间是稳定的还是因渠道而异。输出结果保存到 `analysis/themes-YYYY-MM-DD.md`。

**输出格式：附支持引用的主题报告。** 结果包括一个按频率排序的主题表格、每个主题附带引用和分析的详细章节，以及建议的后续步骤。对于 500 条反馈的分析：10-20 分钟，50,000-100,000 token，$0.25-0.50。

**跨时间趋势追踪。** 定期运行相同分析（月度或季度），比较主题分布变化。"上个月 '慢导出' 是第 3 主题（12% 提及），这个月是第 1 主题（22% 提及）。情况在恶化。"

## 6.4 从访谈记录中提取洞察

访谈提供深度、上下文和未过滤的用户语言。但它们是有噪音的长文档。Claude Code 系统化提取。

**数据准备：** 将访谈记录以纯文本保存到 `data/interviews/`。每条记录以标识符（如 `interview-01-lead-pm.md`）标注。添加 `data/interviews/README.md` 列出每条记录的内容和受访者背景。

> **提示词：访谈综合**
>
> 分析 `data/interviews/` 中的所有访谈记录。提取：跨访谈的关键主题（附带频率和引用的模式）、痛点（用户挣扎的具体任务或工作流）、愿望（用户表达的未满足需求）、意外发现（让你惊讶的引用或模式）。输出保存到 `analysis/interview-synthesis-YYYY-MM-DD.md`。

**跨访谈模式。** 你发现 12 名受访者中有 9 名在无提示的情况下自发提到相同的导出工作流问题。4 名提到"我永远不知道我是否拥有最新数据"。这是信号。

**旅程映射和工作流分析。** 要求 Claude Code 从其分析中生成旅程地图或工作流可视化。

时间：分析 10-15 条记录，15-25 分钟，40,000-80,000 token，$0.20-0.40。

## 6.5 从洞察到行动

综合产生洞察。你需要将这些转化为工程团队可以执行的东西。

**研究方法到产物的流水线**（图 6.4）：原始反馈 → 综合 → 主题 → 用户故事 → 路线图工单。每个阶段提炼并结构化输出，供下游消费。

> **提示词：洞察转行动**
>
> 从 `analysis/themes-YYYY-MM-DD.md` 的反馈综合中，创建可操作用户故事和工单。对每个主题：对应的用户故事（"作为 [用户]，我想 [目标]，以便 [收益]"）、建议的优先级（基于频率 × 严重性 × 战略对齐）、可能需要调查的代码区域（如适用）、建议的下一步（工单/调查/研究）。

**优先级框架。** Claude Code 可以应用 RICE（触达 × 影响 × 信心 ÷ 努力）、价值 vs. 努力矩阵或自定义优先级框架：

> - 高价值 + 低努力 = 立即做
> - 高价值 + 高努力 = 规划项目
> - 低价值 + 低努力 = 如有时间做
> - 低价值 + 高努力 = 不做

Claude Code 生成草稿工单，你可以复制到 Jira 或 Linear。**时间**：映射主题到行动通常需要 15-20 分钟，30,000-50,000 token，$0.15-0.25。
