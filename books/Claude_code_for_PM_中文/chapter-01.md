# 第 1 章 为什么 PM 要用 Claude Code——而不是 Claude.ai

## 1.1 选择合适的工具，每周节省数小时

一位客户报告了你的结账流程中有一个关键 bug。工程团队正全力投入季度路线图。你需要判断优先级：是真正的 bug 还是用户操作失误？影响范围有多大？归哪个团队负责？你等待工程团队切换上下文、调查并反馈。两天过去了，bug 仍然处于待分类状态。

或者：你在代码仓库中打开 Claude Code，问"结账验证是如何工作的？请告诉我 null 支付方式可能在哪些地方遗漏检查"，10 分钟内就得到了带有简明英文解释的文件引用。你整理出包含具体代码位置的调查报告。工程团队确认后直接修复，跳过了发现阶段。总耗时：15 分钟。

这两种场景的区别在于工具选择。大多数 PM 默认对所有事情使用 Claude.ai，因为它更熟悉，然后困惑为什么 AI 没有改变他们的工作流程。网页端处理对话式任务很好，但在 PM 最重要的工作上却力不从心：调查代码库、生成存放于代码仓库的产物、以及构建可复现的流程。

你访问 Claude 有三种方式：Claude.ai 网页界面、API 和 Claude Code。API 是给工程师构建集成用的。Claude.ai 和 Claude Code 都适合 PM，但它们解决不同的问题。选错的代价很高。

Claude Code 是一个具有文件系统访问权限、shell 命令执行能力以及跨会话持久化上下文能力的智能体（agent）。当你的工作涉及文件、代码仓库或可重复流程时，这些能力至关重要——无论你是否专业编程。

选错工具的隐性成本如下。你在 Claude.ai 上花了 30 分钟解释你的代码库背景，让它分析存储在 CSV 中的客户反馈，并请求生成综合文档。它给了你一个深思熟虑的回答。你复制粘贴到 Google 文档中。两周后，你需要对新反馈运行同样的分析。你从零开始：同样的 30 分钟背景说明，同样的手动导出，同样的复制粘贴。你在这个任务上已经花了一个小时，而 Claude Code 使用可复用技能只需五分钟即可完成。

**图 1.1：工作流对比** ——用于理解错误工具选择的隐性成本。左：手动复制粘贴工作流，每次重复上下文、手动导出和重新格式化。右：基于技能的自动化，设置只需一次，后续运行每次 5 分钟 vs 30 分钟。

或者你需要了解结账流程是如何工作的，因为收到了一个关键 bug 报告。你去问工程团队，但他们正在做冲刺规划。你尝试用 Claude.ai，但你无法上传整个代码库，没有上下文，它的回答都是泛泛之谈。你只能等待。工程团队明天才能回复你。bug 又多停留了一天。如果你用了 Claude Code，10 分钟内就能得到一个根本原因假设，并附有具体文件和函数的引用。

工具选择问题的核心在于任务持久性和数据访问。当工作存在于文件中、涉及迭代或发生不止一次时，网页端会强加手动开销，且随着每次重复而累积。Claude Code 通过直接在你的工作区中运行来消除这种开销。

大多数 PM 通过浪费才发现这一点。他们使用 Claude.ai 数周后才意识到，自己一直在向每个对话复制相同的上下文、手动重新格式化输出、重复做本该自动完成的工作。切换成本看似很高：安装、学习命令、理解权限模式。但浪费的成本更高，而且它在无声地累积。

这个决定与技术能力无关。它关乎你的工作是否涉及你会反复接触的文件，还是这只是一个一次性问题。诚实地回答，工具选择不言自明。

## 1.2 Claude Code 如何工作：自主性优于对话

Claude Code 是一个智能体运行时（agentic runtime）。这是技术术语。对产品经理来说，这意味着 Claude 可以行动而不仅仅是回应。当你问 Claude.ai 一个问题，它思考并回答。当你问 Claude Code 一个问题，它思考、读取文件、运行命令、生成产物、检查工作，然后回答。区别在于自主性。

这很重要，因为 PM 的工作很大程度上是跨分散来源的信息综合。需求在 Jira 中，客户反馈在 Zendesk 导出中，实现细节在代码库中，市场调研在收藏的 URL 和杂乱的笔记中。获得一个连贯的答案需要从所有这些来源中提取信息，手动完成会耗费数小时。

Claude Code 在你的项目目录中运行，拥有 Claude.ai 根本不具备的五项能力。理解这些能力，就能明白什么时候你需要的是智能体，而不仅仅是对话。

**图 1.2：五项核心能力** ——用于理解 Claude Code 能做到而 Claude.ai 不能的事情。文件系统读写访问、shell 命令执行、通过 CLAUDE.md 持久化项目上下文、用于并行工作的子智能体（subagent）生成、以及通过 MCP（模型上下文协议）连接外部系统。这些能力支撑了代码库调查、产物生成和可复现工作流。

**文件系统读写访问。** Claude Code 可以打开项目中的任何文件，读取、理解，并写入新文件或修改现有文件。这听起来很基础，直到你意识到 Claude.ai 需要你复制粘贴所有内容。需要在分析反馈时参考 PRD？用 Claude.ai，你要粘贴 PRD、粘贴反馈、请求综合，然后手动保存输出。用 Claude Code，你只需指向 `docs/prd.md` 和 `data/feedback.csv`，它就会直接在代码仓库中生成 `analysis/synthesis.md`。无需剪贴板，无需手动格式化，没有粘贴错误的风险。

对 PM 来说，这意味着调查输出存放在它们该在的地方：在版本控制中，与它们引用的代码并肩，对团队可见。你的代码库分析成为一个 markdown 文件，工程团队可以阅读、评论和更新。

**Shell 命令执行。** Claude Code 可以运行终端命令。对 PM 来说，这主要意味着 git 操作（查看历史记录、比较分支、查看谁改了什么）以及运行那些你不会自己编写但能说明需求的分析脚本。

具体例子：你需要了解某个特定功能是何时引入的、谁参与开发的。你可以请工程团队运行 `git log` 并解读结果，也可以直接问 Claude Code："结账流程的两步验证是什么时候添加的？当时的设计考虑是什么？"它会运行 git 命令，读取相关提交和文件变更，然后告诉你整个故事。无需打扰工程团队。

这使得终端能力通过自然语言变得可用，无需专业知识。

**通过 CLAUDE.md 持久化项目上下文。** 每次打开 Claude.ai，你都从零开始。每次在项目目录中启动 Claude Code，如果存在 CLAUDE.md 文件，它会自动读取：这个文件包含跨所有会话持久化的项目特定上下文。

对 PM 来说，这个文件可能包含产品领域术语表、映射到代码区域的关键用户旅程、团队约定、外部文档链接以及频繁被问及的问题的答案。你只需构建一次上下文，后续每个会话从此都在知情的基础上开始。不再有每次对话都要"这是我们产品的基本情况"的序言。

这就是智能助手和团队成员的区别。团队成员有上下文。拥有维护良好的 CLAUDE.md 的 Claude Code 也有上下文。

**子智能体生成（Subagent spawning）。** Claude Code 可以启动额外的 Claude 实例，并行或在特定专注领域处理子任务。作为 PM，你很少直接调用此功能，但它是复杂工作流的基础。

例如：你要求 Claude Code 准备一份竞争分析，将你的产品与三个竞争对手在功能、定价和定位方面进行比较。它不必按顺序执行，而是可以生成三个子智能体（每个负责一个竞争对手）并行研究，然后将结果汇总到比较矩阵中。这会花费你三个小时的标签页切换时间，缩短到 10 分钟。

这里的成本是真实的（你同时运行多个 Claude 实例），但对于高价值的研究任务，节省的时间往往证明这项投入是合理的。

**通过 MCP 连接外部系统。** 模型上下文协议（Model Context Protocol）让 Claude Code 可以连接到外部工具：Jira、Slack、Figma、数据库、分析平台。这是最新的能力，对于入门来说最不重要，但它消除了破坏工作动力的"导出-导入"循环。

你不必再导出 Jira 工单为 CSV、上传到 Claude.ai、再将结果复制粘贴回去，而是 Claude Code 直接查询 Jira、就地分析、并用发现结果更新工单。工作流保留在一个环境中。第 10 章将详细介绍，但目前只需知道，一旦掌握了基础知识，这是可行且强大的。

这五项能力有一个共同模式：它们消除了思考和行动之间的手动交接。Claude.ai 负责思考，你负责执行。Claude Code 负责思考并执行，然后向你展示结果以供批准。一次性问题两者皆可。可复现的 PM 工作流则受益于自主性。

**智能体循环（Agent Loop）** 是 Claude Code 实际运作的方式。你给它一个目标，它规划方法，执行一个步骤（读取文件、运行命令、生成产物），观察结果，决定继续迭代还是停止，然后向你报告。

**图 1.3：智能体循环** ——用于理解 Claude Code 如何运作。持续循环：接收目标 → 规划方法 → 执行步骤（读取文件、运行命令、生成产物）→ 观察结果 → 决定迭代或停止 → 报告。这个循环是 Claude Code 会话与对话感觉不同的原因——你在观看一个智能体解决问题。

这个循环是 Claude Code 会话与 Claude.ai 对话感觉不同的原因。你观看一个智能体解决问题，看到每一步，并对重要操作保持否决权。起初，详细程度可能感觉过度（"我现在要读取这个文件"），但这建立了信任。你在它提交更改之前看到它在做什么。

对于习惯了要么每件事都需要手动操作、要么像黑盒一样运行的工具的 PM 来说，这种中间地带需要适应。Claude Code 在做工作的同时展示它的工作过程。这是一种让你保持监督的协作模式。

Claude Code 是一个具有工作区访问权限和持久化能力的智能体。它与 Claude.ai 底层使用相同的模型，但能力不同：文件操作、shell 命令和上下文保持。这些能力解决了 PM 的特有问题，因此下一节提供了一个选择正确工具的决策框架。

## 1.3 决定何时使用哪个工具

在 Claude.ai 和 Claude Code 之间选择归结为三个问题：任务是否需要读写文件？你会再次执行这个任务吗？输出是否需要存放在版本控制中？如果任何一个答案是"是"，就从 Claude Code 开始。如果三个答案都是"否"，使用 Claude.ai。

大多数 PM 搞反了。他们默认使用 Claude.ai，因为它更熟悉，然后在需要引用文件或重复工作流时发现各种摩擦。从正确的工具开始可以省去后续的迁移成本。

**图 1.4：工具选择决策树** ——用于为每项任务选择正确的工具。三个问题：是否涉及文件？是否会再次执行？输出是否需要版本控制？任意"是"→ Claude Code。全部"否"→ Claude.ai。不确定时，默认使用 Claude Code 的只读模式。

以下是十二个常见 PM 场景的参考表。建议收藏此页。

| 任务 | 工具 | 原因 |
|------|------|------|
| 起草利益相关者邮件 | Claude.ai | 一次性对话任务，无需文件 |
| 通过代码库访问调查 bug 报告 | Claude Code | 需要阅读代码、git 历史、生成产物 |
| 头脑风暴功能创意 | Claude.ai | 探索性对话，无需持久化 |
| 分析客户反馈 CSV | Claude Code | 文件输入、结构化输出、大概率可重复 |
| 根据文档解释技术概念 | Claude.ai | 快速回答，无需文件操作 |
| 根据 git 历史生成发布说明 | Claude Code | 需要 git 命令、文件输出、每月可重复 |
| 基于研究笔记撰写 PRD 部分 | Claude.ai | 如果笔记在脑海里；如果在文件中则用 Claude Code |
| 了解功能 X 如何实现 | Claude Code | 需要代码库导航和上下文持久化 |
| 竞品功能对比 | Claude Code | 如果可重复；一次性则用 Claude.ai |
| 综合用户访谈记录 | Claude Code | 多文件输入、结构化输出产物 |
| 审查并建议 API 文档改进 | Claude Code | 需要读取仓库中的文档、建议文件修改 |
| 从零创建用户画像 | Claude.ai | 初稿是对话式的；如果数据驱动则用 Code |

模式是：**Claude.ai 用于思考，Claude Code 用于执行**。需要将输出复制到另一个工具的任务，意味着更适合 Claude Code。

**"我还会再做一次吗？"** 这个启发式问题是最快的决策规则。如果你每周都要做这个分析、每月都要创建这个报告、或反复调查这类问题，Claude Code 的设置成本立刻就能回收。即使第一次运行耗时更长（学习命令、结构化提示词、保存技能），第二次运行就快 10 倍。

PM 总是低估重复性。你以为自己只做一次竞争分析，但实际上你在建立一个季度审查流程。你以为自己只写一个用户故事，但实际上你在定义一个你接下来要用 50 次的模板。如果有任何可能你会重复这个任务，偏向 Claude Code，现在就建立可复用的工作流。

相反，如果这真的是一次性的（起草某条特定信息、探索一个模糊概念、获得快速答案），Claude.ai 更快。无需设置，无需学习曲线，只需提问然后继续。启动 Claude Code 进入目录、授权权限、结构化文件输出的开销，只有在你可以摊销这些成本时才有意义。

当你真正不确定时，默认使用 **Claude Code 的只读模式**。使用 `--permission-mode plan` 标志启动，这会阻止任何修改。提出你的问题。如果 Claude Code 的回答涉及"我会读取这些文件并生成这个产物"，那你就用对了工具。如果回答是纯粹的对话，没有文件引用，那你本可以用 Claude.ai。久而久之，这种校准会变成直觉。

一个边界情况：协作性工作。如果输出需要由工程团队审查、存放在代码仓库中或接入 CI/CD 流程，那么无论重复性如何，Claude Code 都是必需的。一份存放在 `docs/` 目录中、与所引用代码并列的调查报告，比 Slack 线程中的同样摘要更有价值。基于文件的产物持久存在、链接到具体提交、与团队工作流集成。而对话式输出会消失。

这个框架帮助你根据任务需求匹配工具能力。两个工具使用相同的模型，区别在于文件访问、命令执行和持久化。大多数 PM 工作涉及文件、迭代和协作。Claude.ai 优雅地处理 30% 的 PM 使用场景，而 Claude Code 处理另外 70% 能真正改变你工作方式的场景。

## 1.4 PM 必须使用 Claude Code 的三大场景

三类 PM 工作需要 Claude Code 的能力。如果你的任务属于其中任何一个，工具选择就已经决定了。

**图 1.5：PM 的三大使用场景** ——用于识别何时 Claude Code 必不可少。代码库调查（无需亲自阅读代码即可了解实现）、上下文产物生成（引用并与代码同步的文档）、以及可复现工作流（编码了重复性 PM 流程的技能）。如果你的工作涉及其中任何一个，Claude Code 就变成了必要的基础设施。

**代码库调查。** 你需要了解某事是如何工作的，但你无法流畅阅读代码，而工程团队有其他的优先事项。这是 Claude Code 对 PM 最经典的使用场景。

客户报告折扣码可以叠加使用，但本不应该。工程团队正全力投入季度目标。你需要判断：这是真正的 bug 还是用户操作失误？如果是真的，有多严重？大概范围是多大？

在代码仓库中打开 Claude Code。问："折扣码验证是如何工作的？请告诉我多个折扣码可能在哪些地方被应用于同一订单。"几分钟内，你就有了文件引用、用简明英文解释的函数，以及关于 bug 的假设。你整理出带有具体代码位置的调查报告。工程团队看到报告，确认问题，直接修复，跳过了通常的发现阶段。PM 总耗时：15 分钟。工程团队节省的时间：一小时的调查加上多次来回澄清。

这个模式不断重复。"认证是如何工作的？""支付失败时会发生什么？""应用在哪里检查订阅状态？"这些问题需要阅读代码、追踪数据流和理解实现细节。Claude Code 通过自然语言查询提供这些能力。

输出存放在 `docs/investigations/` 或相关工单旁。当类似问题出现时，未来的你会受益。未来的团队成员会从文档化的机构知识中受益。调查变成了一个产物，而不是一次丢失的对话。

**上下文产物生成。** 你需要创建一份引用代码库、引用外部研究或需要与代码变更保持同步的文档。文档的价值来自其与代码仓库的集成。

例子：一份引用当前 API 能力并链接到相关文件的 PRD。从 git 历史和 Jira 工单生成的发布说明，用你的产品语调撰写。竞争分析，每季度以相同结构更新，存放在 `research/competitors/` 目录下。随实现演进而保持更新的文档。

Claude.ai 可以写这些文档，但你手动提供上下文、手动保存输出、在事物变化时手动更新。Claude Code 从你的代码仓库读取，通过 MCP 从外部来源获取信息，在原地生成产物，并在你需要下一次更新时重新运行工作流。

第一份季度竞争分析作为技能设置需要 30 分钟。第二次运行只需 5 分钟。第三次、第四次、第五次每次只需 5 分钟。你已经建立了一个可复现的流程，产出结果一致，产物存放在版本控制中，团队可以看到其演变过程。

**可复现工作流。** 你定期做相同类型的任务：每周反馈综合、每月发布规划、季度路线图更新、每个功能的用户故事生成。具体输入会变，但流程是稳定的。

这就是技能（skill）发挥作用的地方。技能是存储在 `.claude/skills/` 中的文档化工作流，Claude Code 按需执行。你只需定义一次流程：所需输入、执行步骤、输出格式、质量标准。然后你调用它："对 `data/customer-feedback-nov-2024.csv` 运行 feedback-synthesis"，Claude Code 执行工作流。

没有技能，你每次都要重复相同的指令（解释分类方案、输出格式、综合深度）。有了技能，工作流被编码下来。你只需关注输入和审核输出。认知负担从"记住我上个月是怎么做的"降到"运行那个东西"。

第 8 章和第 9 章将深入介绍技能。目前，只需知道，如果你做的事超过两次，你就是构建技能的候选者。投入的努力在第三次运行时就能回本。

这三大使用场景（代码库调查、上下文产物生成和可复现工作流）涵盖了 PM 使用 Claude Code 的大部分高价值应用。如果你的工作不涉及这些，Claude.ai 可能就够了。如果你的工作涉及全部三项，Claude Code 就变成了必要的基础设施。

## 1.5 开始之前，了解权衡

Claude Code 解决了真实的问题，但它也带来了 Claude.ai 可以避免的成本和限制。

**学习曲线。** 你需要安装它、理解权限模式、学习基本命令、并培养何时使用哪种能力的直觉。这需要数小时的动手实践。网页界面无需安装，立即可以使用。对于偶尔需要 AI 帮助但很少与代码仓库打交道的 PM 来说，切换成本可能超过收益。

预计第一周会感觉效率低下。你会对命令感到笨拙，忘记在想要只读时设置只读模式，并怀疑这是否值得。到了第二周，如果你在正确的任务上使用它，效率提升开始显现。到了第四周，你会讨厌不得不使用 Claude.ai，因为相比之下它感觉功能受限。但最初几次会话确实会让人感到别扭。

**成本可见性和预算。** Claude.ai 采用订阅定价：固定月费，在速率限制内无限制使用。Claude Code 使用 API 定价（按消耗的 token 付费）。对于探索性较强的 PM 工作，token 消耗可能难以预测。阅读大型代码库、生成多个子智能体或运行复杂的综合任务，可能以订阅定价掩盖的方式消耗你的月度预算。

第 2 章详细介绍了成本管理。简要版：典型 PM 用量预算 $50-150/月，每次会话后用 `/cost` 监控，当成本攀升时使用 `/compact` 减少上下文。如果你对变动成本感到不安或缺乏预算审批权限，Claude.ai 的固定定价可能更可取，尽管有局限性。

**企业部署。** 许多组织限制 API 访问，要求对具有文件系统访问权限的工具进行安全审查，或禁止连接到外部服务。Claude Code 需要 Anthropic API 密钥，并以本地文件权限运行。如果你的组织的安全策略禁止这样做，你只能使用他们批准的官方工具（可能只有网页界面）。

这不是 Claude Code 的限制，而是企业 IT 治理的现实。为获批使用而倡导可以奏效，特别是如果你先在个人项目上展示价值，但要预期一个数周的审批流程。

**Claude.ai 确实更好的场景。** 快速提问。头脑风暴会议。粘贴内容并希望获得对话式反馈的草稿审查。任何你还未定义输出或不需要文件操作的探索性工作。网页界面打开更快，无需考虑权限的心理负担，在任何设备上无需安装即可使用。

如果你发现自己在用 Claude Code 做更适合网页界面的任务，那你正在优化错误的事情。两种工具的存在是因为它们服务于不同的需求。为每个任务使用合适的工具，而不是把一切都强加在你偏好的界面上。

## 1.6 双工具心智模型

两种工具，相同的智能，不同的能力。根据任务需求选择，而不是哪个界面感觉更熟悉。

**Claude.ai：对话式助手。** 你说，它回应。每次对话从零开始。输出留在对话历史中，除非你手动复制到别处。理想用于思考问题、获取快速答案、起草不需要与文件或代码仓库集成的内容。零设置成本，零学习曲线，固定订阅定价。

**Claude Code：具有工作区访问权限的自主智能体。** 你设定一个目标；它读取文件、运行命令、生成产物，并报告结果。对话建立在来自 `CLAUDE.md` 的持久项目上下文之上。输出落在你的文件系统中，受版本控制，对团队可见。理想用于代码库调查、产物生成和可复现工作流。需要安装和学习，变动 API 成本，更高的认知负担。

**决策框架：**

- 这个任务是否涉及文件？→ Claude Code
- 我会再次做这个任务吗？→ Claude Code
- 输出需要存放在代码仓库中吗？→ Claude Code
- 这是一次性对话任务吗？→ Claude.ai
- 我在做没有明确输出的头脑风暴吗？→ Claude.ai

大多数 PM 会经常使用两种工具。Claude.ai 用于 30% 的任务，Claude Code 用于 70%。错误在于对所有人都用 Claude.ai，因为它感觉更容易，然后困惑为什么 AI 并没有真正改变你的工作流程。那些会被改变的工作流，是产物重要的、重复性重要的、与代码仓库集成重要的工作流。

Claude Code 处理这些。这就是本书存在的原因：向你展示如何识别这些工作流、正确地构建它们、并随着时间复利价值。第 2 章介绍设置和安全性。让我们开始吧。
